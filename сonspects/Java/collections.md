### Оцените количество памяти на хранение одного примитива типа byte в LinkedList?

каждый элемент LinkedList хранит ссылку на предыдущий эдемент, следующий за ним и ссылку на данные

- 32 битная система
    - ссылка занимает 4 байта
    - сам объект Node занимает 8 байт
    - 3 * ссылки + Node = 3 * 4 + 8 = 20 байт
    - из-за кратности по 8, получаем 24 байта

    - примитивы упаковываются. Byte = 16 байт
    - 16 = кратность8 (8 байт (заголовок) + 1 байт) = 16 байт

    - так же важно -- значения -128 до 127 -- кэшируются
    
    - итого: 24 байта + 16 байт = 40 байт

- 64 битная система
    - отличия
        - 8 байт -- каждая ссылка
        - 16 байт -- заголовок каждого объекта
    - итого (8 + 8 + 8 + 16) + (24) = 64

### Какие существуют реализации Map?

- Hashtable -- синхронизирован и нельзя пихать null как ключ
- HashMap -- не синхронизирована И можно ключ null
- LinkedHashMap -- упорядоченная реализация хэш-таблицы. упорядоченность засчет дунаправленным связям между элементами. увеличивается память
- TreeMap -- основана на красна-черном дереве. упорядочена. по дефолтку natural ordering, но можно настроить Comparator
- WeakHashMap -- организована с использованием weak-references -- GC автоматически удалит элемент из коллекции при следующей сборке мусора, если на ключ этого элемента нет жестких ссылок


### Как устроена HashMap, сложность основных операций? (Расскажите про принцип корзин)

(что-то до я уже описал, некоторые "очевидные" штуки я пока не пишу)

Разрешение коллизий осуществляется с помощью метода цепочек

- Добавление элемента выполняется за время O(1), потому как новые элементы вставляются в начало цепочки;
- Операции получения и удаления элемента могут выполняться за время O(1), если хэш-функция равномерно распределяет элементы и отсутствуют коллизии. Среднее же время работы будет Θ(1 + α), где α — коэффициент загрузки. В самом худшем случае, время выполнения может составить Θ(n) (все элементы в одной цепочке)

Добавление элементов в объект HasMap
- Сначала ключ проверяется на равенство null. Если это проверка вернула true, будет вызван метод putForNullKey(value)

- Далее генерируется хэш на основе ключа. Для генерации используется метод hash(hashCode), в который передается key.hashCode()

- С помощью метода indexFor(hash, tableLength), определяется позиция в массиве, куда будет помещен элемент.

- Теперь, зная индекс в массиве, мы получаем список (цепочку) элементов, привязанных к этой ячейке.

- Хэш и ключ нового элемента поочередно сравниваются с хэшами и ключами элементов из списка и, при совпадении этих параметров, значение элемента перезаписывается.

- Если же предыдущий шаг не выявил совпадений, будет вызван метод addEntry(hash, key, value, index) для добавления нового элемента

Метод transfer() перебирает все элементы текущего хранилища, пересчитывает их индексы (с учетом нового размера) и перераспределяет элементы по новому массиву.

У HashMap есть такая же «проблема» как и у ArrayList — при удалении элементов размер массива table[] не уменьшается. И если в ArrayList предусмотрен метод trimToSize(), то в HashMap таких методов нет.

### Как работает HashMap при попытке сохранить в него два элемента по ключам с одинаковым hashCode(), но для которых equals() == false?

По значению hashCode() вычисляется индекс ячейки массива, в список которой этот элемент будет добавлен. 

Перед добавлением осуществляется проверка на наличие элементов в этой ячейке.

Если элементы с таким hashCode() уже присутствует, но их equals() методы не равны, то элемент будет добавлен в конец списка.

### Почему нельзя использовать byte[] в качестве ключа в HashMap?

Хэш-код массива не зависит от хранимых в нем элементов, а присваивается при создании массива

метод вычисления хэш-кода массива не переопределен и вычисляется по стандартному Object.hashCode() на основании адреса массива

Так же у массивов не переопределен equals и выполняется сравнение указателей. 

Это приводит к тому, что обратиться к сохраненному с ключом-массивом элементу не получится при использовании другого массива такого же размера и с такими же элементами, доступ можно осуществить лишь в одном случае — при использовании той же самой ссылки на массив, что использовалась для сохранения элемента

### Будет ли работать HashMap, если все добавляемые ключи будут иметь одинаковый hashCode()?

Да, будет, но в этом случае HashMap вырождается в связный список и теряет свои преимущества

### В Java 8 HashMap Treeification Optimization

Количество элементов в одной корзине ≥ 8 (TREEIFY_THRESHOLD).

Общее количество корзин в таблице ≥64(MIN_TREEIFY_CAPACITY).

### А как оно сравнивает ключи в дереве, если хэши равны?
Это очень важный «подводный» вопрос. Чтобы построить дерево, нужно знать, кто «меньше», а кто «больше» (налево или направо идет ветка).
Если `hashCode` одинаковый, Спринг делает так:
1.  Пытается проверить, реализуют ли ключи интерфейс **`Comparable`**. Если да — сравнивает через `compareTo`.
2.  Если нет — использует метод **`tieBreakOrder`**. Он сравнивает названия классов объектов, а если и они одинаковые — использует `System.identityHashCode()`.
