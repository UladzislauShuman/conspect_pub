# JVM

## Введение

что-то о JVM, GC, памяти и прочему

## Содержание
- и так далее

## Какая память есть

Heap -- тут объекты живут
Stack -- локальные переменные, фреймы вызовов. у каждого потока свой.
Metaspace (до 8 Java это PermGen, с -- часть нативной памяти off-heap) -- метаданные классов, статические поля

еще есть
- Code Cache для JIT 
- Direct Memory (off-heap)

## Как делится Heap

- Young Generation -- новые объекты
    - Eden -- где создаются
    - Survivor 0, Survivor 1 -- для тех, кто пережил несколько Minor GC
    - Old Generation -- долгоживущие. пережили много циклов GC

## Где хранятся ВСЕ

Примитив как локальная переменная — на стеке. Примитив как поле объекта — внутри
объекта в куче. Все объекты — в куче. Ссылки на объекты — на стеке (как локальные) или в полях
другого объекта (в куче).

статические поля
- до Java 8 -- Metaspace, PermGen -- часть Heap
- с -- нативная память

это важно тем, что -- при анализе утечек, статические поля не получить из дампа кучи

## Garbage Collector

Как он решает, что можно удалить объект? какие есть стратегии?
- по достижению из корня (статические поля, локальные переменные стека, активные потоки, JNI ссылки)
- есть reference counting -- там могут быть циклические ссылки. первый же способ такого не дает

какие есть
- Serial -- однопоточный. для малышей
- Parallel -- для Young. дефолт до Java 9
- G1 -- дефолт с Java 9
    - делит Heap на регионы, низкие паузы
- ZGC и Shenandoah -- для больших heap, паузы менее 10 мс
- Genertional ZGC -- с java 21

Minor, Major, Full GC
- первый собирает Young Generation
    - это быстро
    - Stop the World, но короткий
- второй -- для Old
    - медленее
- третий -- весь heap и Metaspace
    - плохо, если случается часто

## Виды ссылок

- Strong -- обычная new
    - пока она есть, объект не собереться GC
- Soft
    - собирается GC, когда не хватает памяти
- Weak
    - собирается при следующей сборке, даже если есть память
    - пример -- WeakHashMap
- Phantom
    - для post-finalization очистки
    - редко
