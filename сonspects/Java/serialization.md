## Что такое сериализация и как она реализована в Java?

Это процесс преобразования структуры данных в линейную последовательность байтов для дальнейшей передачи или сохранения. Сериализованные объекты можно десериализовать

стандарты в Java
- стандартная, используя java.io.Serializable
- java.io.Externalizable

позволяет изменять класс, например
- добавить новые поля
- изменить поля из статических в нестатические
- поля из транзитных в нетранзитные (данные, которые можно/нужно передавать в те, которые не учитываются (крч, просто не сохранять те или инные поля. к примеру -- при сериализации User не добавлять пароль))(после десириализации присвоится значение поумолчанию)

Можно ли менять нестатические в статические и нетранзитные в транзитные? -- Да, Но. \
можно просто потерять данные, поэтому нужно быть аккуратным

### Таблица совместимости

Java Serialization Spec (спецификация) говорит нам следующее о том, что можно менять, не ломая чтение старых файлов:

| Изменение | Совместимо? | Что произойдет? |
| :--- | :--- | :--- |
| **Добавить новое поле** | ✅ ДА | При чтении старого файла в новом поле будет `null` (или 0). |
| **Удалить поле** | ✅ ДА | При чтении старого файла данные этого поля просто проигнорируются. |
| **`static` -> `non-static`** | ✅ ДА | Это равносильно **добавлению** нового поля. (В старом файле данных нет -> будет null). |
| **`non-static` -> `static`** | ✅ ДА | Это равносильно **удалению** поля. (Данные из файла игнорируются -> берется текущее значение static переменной). |
| **`transient` -> `non-transient`** | ✅ ДА | Это равносильно **добавлению** поля. (В старом файле его не было -> будет null). |
| **`non-transient` -> `transient`** | ✅ ДА | Это равносильно **удалению** поля. (Данные из файла игнорируются -> будет null). |

Сериализация работает быстрее, чем JSON и XML. ведь это бинарные данные

## Опишите процесс сериализации/десериализации с использованием Serializable.

алгоритм использует Reflection API, который записывает в поток
- метаданные о классе Объекта (имя класса, SerialVersionUID, идентификаторы полей)
- рекурсивно записывает описания super-классов до Object(не вкл)
- примитивные значения полей сериализуемого экземпляра, начиная с полей самого верхнего super-класса
- рекурсивно, но уже объекты, которые поля сериализуемого объекта

Рефлексия -- механизм исследования данных о программе во время ее выполнения (т.е. можно узнать инфу о полях, методах, конструкторах и тп)

Важно -- чтобы избежать цикла в сериализации, ранее сериализованные уже не сериализуются 

Десериализация -- выделяется память под объект и далее поля заполняются данными из потока. Конструктор не вызвыается. Но будет вызвал \ <конструктор без параметров> \<НеСериализуемого класса>. если нет конструктора -- ошибка десериал. .

`ObjectOutputStream` -- для сериализации объектов в поток. методы:
- `void writeObject(Object)`
- `void close()`
- `void flush()` -- очищает буфер и сбрасывает его содержимое в выходной поток

`ObjectInputStream` -- для десериализации
- `Object readObject()`
- `void close()`

## Как изменить стандартное поведение сериализации/десериализации?

реализовать `java.io.Externalizable`.
способ (де)сериализации описывается в `writeExternal` и в `readExternal`

во время Десериал. вызывается `коструктор без параметров`, а потом вызывается `readExternal`

если у сериализуемого класса реализован какой-то метод, то механизм будет испльзовать его, вместо по умолчанию
- `writeObject`
- `readObject`
- `writeReplace` -- дает заменить себч экземпляром другого класса перед записью
- `readResolve` -- дает заменить на себя другой обхект после чтения

## Какие поля не будут сериализованы при сериализации? Будет ли сериализовано final поле?

да, final будет сериализовано 

не будут сериализованы такие поля, которые помечены transient
Пример тех, которые можно так пометить: 
- хранят промежуточное состояние, которое проще вычислить
- ссылка на экземпляр объекта, который не требует сериализации или не может быть сериализован

еще не сериализуются те, которые помечены static. и тогда при десериализации значение static поля не меняется

так же если реализовывать Externalizable, то там можно и static сериализовать. но этого не советуют, так как может будет сложно словить какую-то ошибку

нюанс по поводу final: спокойно сериализуются, но невозможно десериализовать при использовании Externalizable (final поля должны быть инициализированны в конструкторе, и далее они не изменяются. поэтому readExternal ничего потом не сделает).
поэтому нужно применять будет только стандартную сериализацию

## Как создать собственный протокол сериализации?

Для собственного протокола сериализации достаточно реализовать интерфейс Externalizable, у которого есть
- `void writeExternal(ObjectOutput out)`
- `void readExternal(ObjectInput in)`

## Какая роль поля serialVersionUID в сериализации?

используется для указания версии серивализорованных данных

если мы не объвляем его сами, то Java это делает сама. но этот процесс чувствителен ко многим метаданным (количество полей, тип полей, модификаторы доступа полей, интерфейсы и тп)

рекомендуют явно объявлять serialVersionUID, тк при добавлении, удалении атрибутов класса, может поменятся и serialVersionUID, и тем самым в момент выполнения может выброситься `InvalidClassException`

## Когда стоит изменять значение поля serialVersionUID?

нужно изменять, когда внесены в класс несовместимые изменения (пример: удаление атрибута важного). то есть -- я явно хочу запретить чтение старых сохранений

чтобы не допустить автоеатической сериализации, можно переопределить private методы таким образом
```java
private void writeObject(ObjectOutputStream out) throws IOException { 
    throw new NotSerializableException(); 
} 

private void readObject(ObjectInputStream in) throws IOException { 
    throw new NotSerializableException(); 
}
```

## В чем проблема сериализации Singleton?

после десериалзиации мы получим новый объект. то есть -- два объекта Singleton -- противоречие

чтобы избежать
- явный запрет сериализации
- определить readResolve() так, чтобы возвращался замещающий объект, вмместо объета, на котором он вызван

## Расскажите про клонирование объектов. 3 способа

[ссылка на вопрос](https://telegra.ph/10-Rasskazhite-pro-klonirovanie-obektov-3-sposoba-01-05)

проблема -- операторы присвоения передают ссылки, а не создают новые

клонирование -- создаение нового объекта с таким же состоянием

в Object есть protected clone(). нужно его переопределить как public clone(), потом в самом методе вызвать super.clone(). так же класс должен наследовать интерфейс Cloneable (это маркерный интерфейс)

если этого маркера нет, то будет исключение CloneNotSupportedException

интересно, что -- таким образом, объект создается без вызова конструтора.
это решение будет эффективно только если поля клонируемого представляют собой значния базовых типов/их оберток или immurable объектных типов

если же в полях есть ссылочный изменяемый тип, то при коприровании передается ссылка. в таком случае нужно будет клонировать и сам объект поля. что возможно только если он Cloneable

ну и очевидно, что нужно, чтобы по иеррархии и супер-классы были Cloneable

Так же -- следует отказаться от использования объявлений final для полей. ибо нельзя будет изменить значение полей при реализации клонирования

кроме встроенного механизма, можно использовать специализированный конструктор копирования -- в классе описывается конструктор, что принимает объект этого же класса и инициализирует поля создаваемого объекта значениями полей переданного

```java
public Bird(Bird birdOther) { 
    this(birdOther.getName(), birdOther.getAge(), new BirdHead(birdOther.birdHead)); 
} 
```

так же можно воспользоваться Fabric Method-ом (статический метод, что возвращает экземпляр своего класса) И Сериализацию

## В чем отличие между поверхностным и глубоким клонированием?

по умолчанию копирование Поверхностное

как оно проиходит:
- если есть только примитивные типы в классе, то будет совершенно новая копия
- если есть и ссылочные, то копируются ссылки на объекты этих классов. то есть будут иметь одинаковые ссылки

Глубокое же дублирует ВСЮ информацию об объекте
- нет необходимости копировать отдельно примитивы
- все члены ссылочного типа в оригинальном классе должны поддерживать клонирование
- если какой-то член класса не поддерживает копирование, то в методе нужно создать новый экземпляр и скопировать каждый его член со всеми атрибутами в новый объект

## Какой способ клонирования предпочтительней?

самый безопасный это использование специализированного конструктора копирования
- нет ошибок наследования (не нужно думать, что у наследников появяться новые поля, что не будут склонированы через clone())
- поля для клонирования указываются явно
- можно клонировать даже final поля


## Почему метод clone() объявлен в классе Object, а не в интерфейсе Cloneable?

метод clone() объявлен в классе Object с указанием модификатора native, чтобы обеспечить доступ к стандартному механизму поверхностного копирования

и так же он объявлен как protected, чтобы было нельзя вызывать этот метод у не переопределивших его объектов

А сам Cloneable -- это маркер, просто чтобы обозначить сам факт, что объект готов быть клонированным. 

вызов переопределенного метода clone() у не Cloneable объекта вызовет исключение CloneNotSupportedException
