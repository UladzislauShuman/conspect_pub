## Ответы

### История
- Когда?
    - вышли Kotlin Coroutines в 2018 году
- проблема
    - основная проблема -- Callback Hell
        - если нужно было
            - запросить id пользователя из БД
            - потом по нему дернуть API
            - результат сохранить в кеш
        - то получим гору вложенных функций обратного вызова
- чего они хотели
    - императивный стиль и чтобы все выглядело просто
    - легковесности. чтобы кучу задач запустить на телефоне
    - независимость от версии Java: Android до сих пор сидит на старых версиях Java. нужно решение, которое даст удобную многопоточность без обновления JVM

### Задача о пополнении и снятии со счета

- Условие 
    - Банковская система, у пользователя единый счет. В один момент времени происходит два события
        - пополнение счета
        - снятие со счета
    - то есть -- обе операции считывают текущее значение баланса, проводят вычисления и записывают новый результат обратно

- проблема
    - Race Condition
    - описание: если одновременно произойдут два считывания, то при записи будет записано одно из значений, а не их сумма
        - либо счет + деньги 
        - либо счет - деньги
        - другой вариант потеряется (да -- ты можешь потерять зарплату)


- ответы на вопросы 
    - почему обычные переменные не обеспечат корректности?
        - если мы делаем `var balance: Int, balance += 500`, то -- это 3 операции
        - "по умолчанию" они не атомарны
        - и так же -- нет видимости по умолчанию -- согласно JMM, потоки могут локально кешировать переменные в локальной памяти процессора. 
        без спец синхронизации, один может поменять, а другой просто этого не увидеть 
    - что можно
        -  классические потоки
            -  ставит WAITING и переключает контекст -- жирно
        - atomic-переменные, 
            - операции реализованы по принципу Compare-and-swap
            - если все еще 1000, то запиши 1500. иначе -- повтори операцию. 
            - проблема очевидна -- когда на счету после операции другой осталось 1000$, то как это воспринимать? как новое состояние или все тоже?
        - Корутины
            - это легковесные потоки, что работаю поверх JVM
            - Mutex
                - аналог ReetrantLock
                - это suspend-функция.
                    - если корутина не может получить замок, то она не блокирует поток. она как бы "засыпает". контекст не переключается -- эффективно
            - просто в один поток все сделать

- код 
    ```kotlin
    import kotlinx.coroutines.* // Библиотека для работы с корутинами
    import kotlinx.coroutines.sync.Mutex // Специальный "замок" для корутин
    import kotlinx.coroutines.sync.withLock // Удобная обертка для Mutex
    import kotlin.system.measureTimeMillis

    class BankAccount(initialBalance: Double) {

        var balance Double = initialBalance
            private set 

        // Mutual Exclusion -- взаимное исключение
        // неблокирующий замок. 
        // если одна корутина заняла Mutex, то вторая не стопит поток ОС, а "засыпает" (suspend)
        // освобождает поток для других задач
        private val mutex = Mutex()

        /**
        * suspend
        * ключеове слово. обозначает функцию "которая может быть приостановлена"
        * под капотом: превращает эту функцию в конечный автомат
        * не блокиурет поток исполнения, а сохраняет свое состояние и "отпускает" поток
        * suspend функцию можно либо из другой suspend-функции вызвать, либо в корутине
        */
        suspend fun deposit(amount: Double) {
            mutex.withLock {
                val current = balance
                delay(10) // имитация задержки
                balance = current + amount
                println("Зачисление $amount | Текущий баланс $balance")
            }
        }

        suspend fun withdraw(amount: Double) {
            mutex.withLock {
                if (balance >= amount) {
                    val current = balance
                    delay(10)
                    balance = current - amount
                    println("Снятие $amount | Текущий баланс $balance")
                } else {
                    println("Недостаточно средств для снятия $amount")
                }
            }
        }
    }

    /**
    * runBlocking -- связывает блокирующий поток с корутиной
    * создает корутину и блочит поток, пока все задачи внутри не завершатся
    * обычно используется редко
    */
    fun main() = runBlocking {
        val account = BankAccount(1000.0)
        println("Начальный баланс: ${account.balance}")

        // замеряет время работы кода 
        val time = measureTimeMillis {

            /**
            * launch -- создает корутину. работает в фоне. не дожидается результата
            */
            val job1 = launch {
                repeat(5) { // имитируем 5 зарплат
                    account.deposit(100.0)
                }
            }

            val job2 = launch {
                repeat(5) {
                    account.withdraw(200.0)
                }
            }

            // заставляет main корутину подождать завершения джобы (нашей дочерней корутины)
            job1.join()
            job2.join()
        }
        
        println("Финальный баланс: ${account.balance}")
        println("Все операции заняли: ${time}мс")
    }
    ```


### Корутины. Пример

- кухня. поток -- это повар. корутина -- рецепт.
- в классической многопоточности: если повар хочет поставить макароны вариться и пойти резать лук, то ему нужен второй поток.
- в корутинах же -- повар ставит макароны варится (пока вода закипит), а дальше идет резать лук. 

### Сравнение с Project Loom. почему Coroutine не перешли в Java?
Project Loom -- системное решение на уровне JVM.

JVM сама понимает, когда поток заблокировался и отцепляет его от реального потока

Coroutines -- это по сути просто куча объектов и колбеков. все происходит на уровне языка и компилятора. JVM даже не знает о них 

а что из них вообще лучше?
- Loom
        - он проще. не нужно писать ключевое слово suspend. старый блокирующий код Магическим образом начинает работать быстрее
- Coroutines
    - дают гибкий контроль над жизненным циклом (Structured Concurrency)
    - то есть, можно легко сделать такой flow: "если эта задача отменилась, отмени и все дочерние запросы к БД"
    - работают там, где нет JVM

почему Java не взяла корутины?
- Java думает об обратной совместимости. она не хочет такого сценария: вышла suspend И все стали переписывать свой код и обновлять под suspend
- прозрачность: разработчики хотели, чтобы разработчик не думал "блокирующая операция это или нет", Loom делает любую такую операцию дешевой
- Java решила поменять саму JVM, а не добавлять новое слово, чтобы строить стейт-машину

почему корутины эффективные?
- корутина -- это просто объект. он занимает мало места
- переключение контекста в корутине -- это просто смена указателя в памяти

почему Kotlin выбрали именно такую реализацию?
- это им позволило внедрить асинхронность в язык без изменения самой State Machine в Java
- и в итоге мы получили мощную асинхронность на старых версиях Java

Когда бы я выбрал
- Project Loom
    - легаси-проект на чистой Java
    - прямое взаимодействие с блокирующим IO -- старые драйверы БД что нельзя заменить
    - у меня весь стек это Java
- Coroutines
    - мультиплатформа . ибо Loom это только для JVM
    - удобно управлять группами потоков
    - нужна реактивность
    - UI разработка 

### Structured Concurrency

- что это
    - это парадигма программирования, в которой время жизни асинхронных задач строго ограничено определенной областью видимости (scope)
    - а быть точно
        - иерархия: каждая новая корутина запускается только внутри определенного CoroutineScope
        - связанный жизненный цикл: дочерние корутины не могут пережить своих родителей. родитель умер -- умерли все дети
        - родительская корутина не считается завершенной до тех пор, пока не завершаться все ее дочерние корутины

- зачем это?
    - до появлния было Un-структурированно). это приводило к проблемам
        - утечка ресурсов: задача могла продолжать выполеяться в фоне, даже если ее результат уже не нужен
        - чтобы остановить группу потоков, нужно было самому держать ссылки на каждый из них и вызвать interrupt
        - если один из множества потоков падал с исключением, то это могло остаться незамеченным или не приводило к остановке остальных задач

### Как это все работает вообще?

как вообще работает компиляция в котлин, поверхностно

ступени компиляции
- FrontEnd
    - разрезает текст на токены (слова, знаки и пр)
    - строит AST (Abstract Syntax Tree) -- дерево синтаксического разбора
    - проверяет типы
- Intermediate Representation (IR)
    - чертеж моей программы
    - это нужно, чтобы Kotlin был мультиплатформенным
    - frontend один для всех, а IR можно сделать и код в Java, и в JS, и в Native
- Backend
    - компилирует IR в конечный результат (байт-код jvm, например)

тем самым, напомню -- корутины это как результат генерации кода на этапе компиляции, а не работа с JVM как в Project Loom

расмотрим пример кода 
```kotlin 
suspend fun doWork(id: Int): String {
    println("Start $id")
    delay(1000) // Точка приостановки 1
    println("Middle $id")
    delay(2000) // Точка приостановки 2
    return "Done $id"
}
```
далее в генерации происходят такие навароты
- изменяется сигнатура (CPS - Continuation Passing Style)
    - в конец параметров каждой suspend-функции добавляется Continuation
    - Contination -- это объект, который знает, как продолжить выполнение функции
    - также меняется возвращаемый тип функции -- теперь он Any? (тут скрывается либо реальный результат, либо маркер COROUTINE_SUSPENDED)
- создается State Machine -- анонимный класс внутри фунции, который отслеживает текущий шаг выполнения через поле label

как это выглядит в итоге
```kotlin
public Object doWork(int id, Continuation completion) {
    // Внутренний класс для хранения состояния
    class MyStateMachine extends ContinuationImpl {
        int label = 0;
        int savedId;
        Object result;

        @Override
        public Object invokeSuspend(Object result) {
            this.result = result;
            return doWork(0, this);
        }
    }

    MyStateMachine sm = (completion instanceof MyStateMachine) 
        ? (MyStateMachine) completion 
        : new MyStateMachine(completion);

    switch (sm.label) {
        case 0:
            System.out.println("Start " + id);
            sm.savedId = id;
            sm.label = 1;
            // Вызываем delay и передаем машину состояний как Continuation
            if (DelayKt.delay(1000, sm) == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED;
        case 1:
            // Восстанавливаем локальные переменные
            id = sm.savedId;
            System.out.println("Middle " + id);
            sm.label = 2;
            if (DelayKt.delay(2000, sm) == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED;
        case 2:
            id = sm.savedId;
            return "Done " + id;
    }
    return null;
}
```

- компилятор разразает функцию в каждой точке, где стоит suspend. каждый такой кусок становится отдельным состоянием (case в switch) в State Machine
- все локальные переменные, что используются после точки приостановки (в нашем примере это id), переносятся из стека в поля объекта машины состояний. позволяет запомнить состояние, когда потоки освобождаются
- suspension (приостановка) -- когда вызывается другая suspend функция (delay), то текущая функция передает ей сам себя (sm) в качестве аргумента Continuation и возвращает маркер COROUTINE_SUSPENDED. поток в это время не блокируется, а просто уходит выполнять другие задачи
- возобновление (resuming) -- как только фоновая работа завершена, вызываетя метод у объекта continuation.resumeWith -- в тут в контекте так получается, что нашей State Machine. наша State Machine снова вызывает нашу исходную функцию, но уже благодаря сохраненному состоянию и label мы Продолжаем ее выполнять с установленного места 

### Диспетчеры

- что это?
    - компонент, что определяет, какй поток или пул потоков будет использовать корутину для Свовего выполнения
    - то есть -- коркутина -- это задача, диспетчер -- это планировщик, что назначает исполнителя, а поток -- это исполнитель

- зачем?
    - задачи бывают разные
        - кто-то требует много CPU / производительности в принципе (расчеты, графика и тп)
        - где-то мы в основном ждем чего-то (IO, сетевые запросы и тп)
        - и где-то мы хотим, чтобы наш код выполнялся в одном потоке (UI)
    - и это задача диспетчера -- эффективно управлять каждым согласно его "условиям"

- виды диспетчеров
    - `Dispatchers.Main`
        - только для легких задач и для задач в одном потоке. то есть UI
        - если запустить тяжелую задачу, то интерфейс залипнет
    - `Dispatchers.Default`
        - работает в общем пуле потоков. количество потоков обычно равно количеству ядер
        - как раз для задач с большим потреблением CPU
    - `Dispatchers.IO`
        - в специальном пуле потоков для операций ввода/вывода
        - IO-операции
        - тут уже Потоков может создаваться куда больше, чем в Default
    - `Dispatchers.Unconfined`
        - запускает корутину в текущем потоке До первой остановки. после возобновления она может продолжить работу в совершенно другом потоке
        - обычно не рекомендуют для общего использования. подходит для задач, что не работают с общими данными и не грузят процессор

- как это работает?
    - я вызвал `launch(Dispatcher.IO)`
    - моя корутина передается диспетчеру
    - диспетчер ставит ее в очередь своего пула потоков
    - освобождается поток в пуле -- диспетчер вызывает метод `run()` у задачи и корутина начинает выполняться в этом потоке
    - если корутина дошла до `delay` или другого `suspend` вызова. она отпускает поток
    - когда данные получены, диспетчер заново находит свободный поток из очереди и возобновляет работу корутины

- это вообще эффективнее Java потоков?
    - в Java мы бы создали new Thread или ExecutorService, а в корутинах диспетчеры используют общие пулы. млн корутин могут использовать всего 10 Реальных потоков
    - корутина занимает поток только тогда, когда реально работает, и отдает, когда ждет

### CoroutineContext

- что это?
    - это набор элементов (чем-то напоминает Map), что определяет поведение и среду выполнения корутины
    - элементы контекста корутины
        - Job -- это элемент, который представляет саму корутину как фоновую задачу с жизненным циклом
            - можно использовать для отмены (`job.cancel()`) или ожидать ее завершения (`job.join()`)
            - ну и состояния
                - `isActive`
                - `isCompleted`
                - `isCancelled`
        - CoroutineDispatcher -- определяет поток или пул потоков
        - CoroutineName -- имя корутины для отладки
        - CoroutineExceptionHandler -- обработчик исключений (типо try catch на уровне контекста)
    - и как это интересно работает
        - мы можем комбинировать эти элементы с помощью плюса(+)
            - `Dispatchers.IO + CoroutineName("MyCoroutine")`

- Еще раз об иерархии и наследовании в нашем Structured Concurrency 
    - Новая корутина наследует контекст родителя
    - Job новой корутины становиться Ребенком родительского Job
    - Если вы отменяете родительский Job, то отменяются и дочерние
    - Родитель не завершается, пока не завершатся все его дети
    - Но
        - когжа вы указываете новую область (GlobalScope) или явно передаете новый Job в параметры launch, то иерархия рушится

### Как тут исключения обрабатывать вообще

- как распространяется ошибка
    - в дочерней корутине наступила ошибка
    - она перешла Родителю
    - родитель отменил 
        - себя
        - всех своих детей
    - и далее ошибка идет выше
    - и как видно -- это может привести к тому, что Все приложение уронилось

- решение -- SupervisorJob или supervisorScope
    - SupervisorJob -- это Job, что игнорирует ошибки дочерних Job. то есть -- ничего не отменяется как сказано выше
    - supervisorScope -- блок окда, что создает временную область видимости с SupervisorJob. отличное место для запуска нескольких независимых задач (загрузка картинок: одна не скачалась, но все остальные должны)

- CoroutineExceptionHandler
    - чтобы поймать исключение, что не было обработано в `try-catch`
    - важно: работает только если установлен в корневой корутине или CoroutineScope. если добавить его в дочерний launch, то он будет проигнорирован, ибо ошибка все равно к родителю пойдет

- о разнице launch и async -- механизм ошибки зависит от билдера
    - launch -- ошибка для него критическая. если не поймать -- все ляжет
    - async -- ошибку инкапсулирует в Deferred и выстрелит только если вызвать `.await()`. Но если async в обычном scope запущен (не супервизор), то ошибка дойдет все равно до родителя и без await

### Наташа, Четвертая корутина, Отмена!

- а в чем особенность
    - в старых Java-потоках, которые можно попытаться остановить принудительно, корутину нельзя просто так отменить извне. Корутинный код должен как бы Сотрудничать, чтобы быть отменяемым
    - Если корутина занята И не проверяет свой статус отмены, то она продолжит свою работу

- как это сделать?
    - использовать встроенные suspend-функции по типу delay, yield, withContext, которые сами по себе отменяемые и в которых есть проверка на статус отмененности. если они у вас часто вызываются, то все будет работать в принципе
    - можно самому проверять флаг isActive
    - использовать `ensureActive()`. ее можно ставить в начале каждой Итерации цикла
    - yield() -- не только отменяет корутину в случае если, но и уступает другим путем временного освобождения потока

- о CancellationException
    - с помощью него и реализована отмена
    - это просто своего рода сигнал, что "хватит"
    - прикол еще в том -- оно не приведет к падению приложения. оно игнорируется стандартным обработчиком
    - но если использовать `catch(e: Throable)`, то можно и поймать

- Неотменяемый код
    - если хочется, чтобы какой-то блок кода Точно выполнился до конца (закрытие ресурсов в finally) даже если корутину отменили, модно воспользоваться `withContext(NonCancellable)`

### Flow

- что это?
    - это асинхронный поток данных, что может выдавать несколько значений последовательно по мере их готовности
    - особенность
        - если List -- это ведро, то Flow -- это водонапорный кран
        - Flow не хранит все данные в памяти Сразу

- Cold Flows
    - является холодным -- то есть не начинает выполнения до тех пор, пока не вызовете терминальный оператор 
    - на каждый вызов collect поток данных запускается заново для конкретного получателя

- основные компоненты или даже этапы
    - Producer -- создает поток и передает данные с помощью `emit()`
    - Intermediary -- операторы трансформации (map, filter и тп). тоже холодные
    - Consumer --  вызывает терминальные операторы (collect, например). они запускают выполнение потока

- почему это эффективнее обычных списков?
    - Flow построен на корутинах и тем самым -- ничего не блокируется при ожидании
    - можно управлять контекстом и с помощью оператора `flowOn(Dispatchers.IO)` переключить выполнение тяжелых операций на фоновые потоки , а сбор данных при этом будет из основного потока
    - есть возможность отмены -- кооперативная отмена корутин есть (если корутина, в которой вызывается collect, отменяется, то работа потока также прекращается)

- горячие потоки (StateFlow и SharedFlow)
    - они хранят состояние или передают данные, даже если у них нет подписчиков. они тем самым горячие
    - StateFlow -- хранит последее состояние и передает его новым подписотам. можно часто встретить в Android для обновления UI
    - SharedFlow -- может транслировать события всем подписчикам (пример -- уведомления). не хранит состояние по умолчанию, хотя и имеет параметры буфера

### Отличие launch и async

- Сравним по
    - возвращаемое значение
    - сценарий
    - обработка ошибок

- launch
    - работает по приницу ЗапустилИЗабыл. возвращает Job, что дает возможность управлять жизненным циклом. но не возвращает результат вычислений
    - когда не сильно то и нужен результат выполнения задачи (сохранить в бд, записать в логи и тп)
    - исключения пробрасываются в родительский контекст, что аналогично обычным потокам. непойманая ошибка может привести к падению приложения

- async
    - нужен, когда нужно получить результат вычислений. возвращает объект Deffered\<T>  как аналог Future или Promise. чтобы получить значение -- await()
    - для параллельных вычислений -- самое то (выполнить несколько запросов одновременно и объединить результаты)
    - исключение будет инкапсулированно в Deferred и выстрелит только если использовать await(). но в рамках нашей структурированной конкурентности ошибка в async все равно уведомит родителя и может отменить соседние корутины

### Как бы ты обработал поток данных из нескольких сервисов? (`zip`, `combine` и `merge`)
у меня просто была такая задача на собеседовании. она тривиальная достаточно, но тем не менее

```kotlin
import kotlinx.coroutines.*
import kotlinx.coroutines.flow.*

fun getServiceAFlow() = flow {
    emit("Data A1")
    delay(100)
    emit("Data A2")
}

fun getServiceBFlow() = flow {
    emit(10)
    delay(200)
    emit(20)
}

fun main() = rubBlocking {
    val flowA = getServiceAFlow()
    val flowB = getServiceBFlow()

    // попарно объединяет. ждет, пока каждый выпустит соответсвующие элементы
    flowA.zip(flowB) {
        a,b -> "$a + $b"
    }.collect {
        println(it)
    }

    // объединяет последние выпущенные значнеия. сработает всегда, когда кто-то выпустит значение
    flowA.combine(flowB) {
        a, b -> "$a and $b"
    }.collect {
        println(it)
    }

    // сливает потоки 
    merge(flowA, flowB.map { it.toString() })
        .collect {
            println(it)
        }
}
```

- zip -- требует наличия пары. если один поток завершается раньше, то оставшиеся элементы второго будут отброшены
- combine -- отлично для случаев, когда нужно иметь актуальное состояние из нескольких источников. он объявляет результат сразу, как только приходит новое значение от любого сервиса, используя последнее известное значение от другого
    - пример задачи: обновление значка уведомлений или статуса пользователя в реальном времени
    - фильтрация поиска
    - корзина товаров
- merge -- не связывает данные межлу собой, а как объединение нескольких потоков
