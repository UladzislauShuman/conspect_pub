### Проблема

#### До
Как раньше было -- одно ядро. то есть -- многопоточности как бы и -- не было. как написал -- так и шло. 
as-if-serial семантика - гарантия того, что резульатт выполнения программы будет таким, как если бы все инструкции выполнялись строго по порядку написанному

#### Появление
мы хотим, чтобы программы работали быстрее, ну и сделали оптимизации на трех уровнях

- Компилятор (JIT, javac) -- может поменять строчки кода местами (Instruction scheduling)
- Процессор -- out-of-order execution или спекулятивные вычслиения. процесс не ждет, пока выполнится долгая операция, он начнет уже выполнять следующую за ней инструкуцию
- память и кеши - ядра больше не хотял напрямую в оперативку, а уже в свой Локальный кэш(L1, L2) и спец буферы записей

#### Суть
наша гарантия as-if-serial работает только для одного потока в изоляции
- Visibility 
    - поток пишет x = 5 -- это не летит в общую оперативку, а зранится в локальном кэше ядра. а второе ядро -- либо идет в кэш, либо в оперативку -- и видит старое значение
- Reordering
    - стандартный пример
    ```java
    // Сначала
    x = 0, y = 0

    // Thread 1
    x = 1
    int r1 = y

    // Thread 2
    y = 1
    int r2 = x
    ```
    - сможем ли мы получить r1 = 0, r2 = 0?
    - Да, как бы странно это не показалось
    - компилятор или процессор видит, что в Thread 1 переменные x и y никак не связаны друг с другом. запись в x стоит дорого, а чтнение y можно сделать быстро. процессор меняет их метами под капотом: сначала y получает 0, а потом пишет x = 1. и тоже самое в Thread 2. в итоге -- оба нули

#### А Кроссплатформенность?

разные процессоры могут иметь разную аппаратную модель памяти
- кто-то более строгий и запрещает большинство переупорядочиваний, а кто-то менее -- лишь бы сэкономить энергии

а Java хочет быть "Write Once, Run Anywhere". поэтому важно, чтобы программа работала в первую очередь одинаково на разных системах


### Решение

JMM как решение проблемы

- что это
    - это контракт, это абстракция между Нами и Железом
    - "я гарантирую, что если ты будешь использовать этот контракт Правильно, то все будет работать так, как ты хочешь, внезависимости от платформы"

- как JVM выполняет этот контракт под капотом
    - как JVM заставляет физический процессор перестать переупорядочивать инструкции и сбрасывать кэши
    - для этого JVM использует аппаратные барьеры памяти (Memory barriers)
        - это специальные низкоуровнивые инструкции процессора, что: когда процессор натыкается на такой барьер, то он понимает "дальше оптимизировать нельзя. нужно завершить все предыдущие записи в память и обновить кэш"
        - типы базовые типа барьера
            - LoadLoad -- запрещает переупорядочивать чтения
            - StoreStore -- запрещает переупорядочивать записи
            - LoadStore -- чтение До барьера не может быть переупорядочено с записью после
            - StoreLoad(самый тяжелый) -- запись до барьера должна быть видна всем процессорам до того, как начнется чтение после барьера. он и решает проблему с x и y выше
        - JVM сама решает, какие именно барьеры расставить в скомпилированном машинном коде в зависимости от архитектуры процессора

- а как разрабу этим всем пользоваться?
    - для "Эй, мне вот тут нужна гарантия JMM!" мы используем ключевые слова
    - `volitile` -- решает проблему Видимости и частично Переупорядочивания
    - `synchronized` и классы из `java.util.concurrent.locks` -- решает проблемы Атомарности и Видимости 
        - когда поток выходит из блока помечанного этим словом, он гарантированно сбрасывает все изменения в основную память
        - а когда другой поток входит в этот монитор, то он гарантиованно достает самые свежие данные 
    - `final` -- дает гарантию безопасной инициализации
        - если объект создан и у него поля помечаны этим ключевым словом, то JMM гарантирует, что любой другой поток увидит эти поля полностью инициализированными (без final, другой поток мог бы увидеть объект наполовину созданым)

- а почему бы не сделать все безопасным по умолчанию? 
    - производительность, ведь -- барьеры памяти это очень медленно (в десятки и сотни раз медленнее)
    - JMM дает нам выбор -- по умолчанию все быстро, но если надо -- делай

### Основные концепции (Data Race, Happend-Before)

- Data Race
    - из определение JMM, гонка происходит при трезх условиях одновременно
        - два потока обращаются к одной и той де переменной
        - хотя бы одно из обращений -- запись
        - эти обращения не связаны отношением Happens-Before
    - Правило -- если есть Data Race, то JMM ничего не гарантирует. если его нет -- программа работает предсказуемо

- Happens-Before
    - это отношение между двумя действиями (Запись А и Чтение Б)
    - пример
        - если Действие А Происходит-До Б, то JMM гарантирует
        - "Все, что было изменено в памяти До и Во время дейсвтия А, будет видимо потоку, выполняющему действие Б, и действие Б выполняется строо после А"
    - важно: Это понятие не означет, что инструкции физически не будут переупорядочены процессором. процессор может менять их как хочет, НО только если это не нарушит конечный результат, что гарантируется Happens-Before

- как получить Happend-Before
    - как связать два потока этим правилом -- нам дает JMM список правил
    - Program Order Rule (Правило одного потока) 
        - любое действие в рамках одного потока Происходит-До любого действия, что написано после него в коде
    - Monitor Lock Rule
        - освобождение монитора Происходит-До захвата того же самого монитора другим потоков
    - Volatile Variable Rule
        - запись в volatile переменную Происходит-До любому последнему последующему чтению той же самой переменной
    - Thread Start Rule
        - Thread.start Происходит-До первому действию внутри нового потока
    - Thread Join Rule
        - Последнее действие в потоке Происходит-До выхода из метода Thread.join() в другом потоке (который ждал завершения)
    - Транзитивность
        - A h-b B and B h-b C => A h-b C

    - C hb D

### Про long, double 

JMM гарантирует, что чтение и запись любых переменных, причем даже без volatile -- атомарны. то есть ты не можешь прочитать половину значения, но все таки есть и исклбючения -- long и double

они занимают 64 бита, что в два раза больше, чем спецификация JMM, что написана во времена 32-битных процессоров. и эта спецификация разрешает запись 64-битного значения на две операции 32-бита

то есть да

если long x = ... // Thread 1

и long x = ___ //Thread 2

то может получится фигня при чтении из Thread 3

такая фигня и называется -- Word Tearing (буквально разрыв слова)

и чтобы этого избежать в случае long и double, то нужно использовать volatile

хочеться уточнить6 что -- в современных 64-бит JVM такого как бы и не встречается, но в спецификации есть. вероятно это связано с тем, что новые процессоры -- они в принципе за один такт пишут 64 бита

почему они кстати так сделали?
- потому что иначе бы мат операции были бы слишком медлинными

кстати
- ссылки то в 64 бита тоже 64 бита. а они рвутся?
- JMM тут все таки сделала исключение и сказала, что чтение и запись ссылок атомарна всегда



### final

- проблема
    - создание объекта это не атомарная операция
    - это как минимум три шага
        - Allocate -- выделить память 
        - Init -- конструктор
        - Publish -- присвоить ссылку переменной
    - может случиться Reordering того же 2 и 3, ну и ссылка уже будет не null, но поля внутри будут null

- решение 
    - определенные гарантии JMM дает с помощью слова final на полях объекта
    - в конце конструктора, где инициализируются final поля, JVM ставит специальный барьер памяти (Freeze Barrier)

- Freeze Barrier
    - запрещаю публиковать ссылку на этот объект, пока все final поля не будут физически записаны в память и видны всем

- гарантия
    - если другой поток увидел ссылку на объект, он гарантированно увидит правильно инициализированные final поля этого объекта

- поэтому иммутабельные объекты потокобезопасны и их не нужно синхронизировать 

- кстати важно
    - а что если final поле -- это ссылка на массив или другой объект? JMM гарантирует, что я увижу внутренности этого массива?
    - Да. final распространяется вглубь
- еще важно
    - final работает Только в том случае, если ссылка this не утекла из конструктора до его завершения
    - пример, когда плохо
    ```java
    class EventListener {
        final int id;
        
        EventListener(EventBus bus) {
            this.id = 100;
            bus.register(this); 
        }
    }
    ```

- для котлина
    - val превращает в private final
    - тем самым data-class-ы с val полями -- они имутабильны и нам не нужно делать volatile и synchronized, чтобы передавать data class между потоками. уже все за нас сделано!

### Железо. MESI и Store Buffer 
- важно понимать
    - процессор работает быстро
    - а поход в память в десятки раз дольше
    - поэтому к ядру прикручивают кэш

- Протокол MESI (когерентность кэшей)
    - ядра процессора общаются друг сдругом, чтобы синхронизировать l1,l2 кэши. то есть: если ядро 1 меняет переменную, то она шлет сигнал "invalidate" другим ядрам, чтобы они стерли значения, НО -- ждать ответа это долго
    - общение происходит по специальной шине
- Store Buffer (буфер записи) 
    - чтобы не ждать, ядро 1 кладет новое значение в свой локальный Store Buffer и продолжает работать. И значение застревает там и не попадает в кэш, отсюда и проблема невидимости
- Invalidate Queue (очередь инвалидации) 
    - ядро 2 получает сигнал "сотри старое значение", но ему реально некогда. и оно кладет сигнал в очередь и продолжает читать старые данные из кэша

и как же тогда работает volatile?
- когда он видит такой барьер памяти, то он принудительно сбрасывает весь Store Buffer в кэше, и принудительно обрабатывает всю invalidate queue. тем самым железо синхронизируется

### Может ли Формула 1 быть безопасной?
хз честно, не смотрю, но

Data Race это плохо, но ради в производительности он есть даже в исходниках Java
- String.hashCode()
    - в String хэш вычисляется лениво (при первом вызове) и кэшируется в поле hash
    ```java
    private int hash; // По умолчанию 0

    public int hashCode() {
        int h = hash;
        if (h == 0 && value.length > 0) {
            // Вычисляем хэш на основе массива символов value
            h = isLatin1() ? StringLatin1.hashCode(value) : StringUTF16.hashCode(value);
            hash = h; // Data Race: запись без синхронизации!
        }
        return h;
    }
    ```
    - тут есть классика Data Race -- оба увидели h == 0 и вдвоем начали считать
    - но почему они так сделали?
    - ну потому что -- строка то одинаковая будет, ну и все равно получится один и тот же результат. следовательно -- зачем заниматься такой вот дупней?
    - то есть, вычисление хэша строки Идепотентно (дает один и тот же результат для одной и той же строки)

- еще пару причин, почему это не ломается
    - как мы сказали выше, 32 бита атомарно читаются и записываются. это я к тому, что у нас hash это int -- который 32 бита
    - массив value объявлен как final. поэтому не получится ситуации когда "при инициализации массив на половину только полон, ну и мы уже начали считать ему hash"
    - ну и индемпотентность вычисления хэша

fun fact 
- есть же такая строка, хэш которой будет равен 0
- ну и следовательно -- каждый раз пересчитывается 
- и разрабы поняли -- шанс такого очень мал, ну и не решили они ради такого редкого случая ломать быстроту работы системы в общем
