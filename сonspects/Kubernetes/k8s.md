## 1. Архитектура: основные компоненты

![фото](https://kubernetes.io/images/docs/kubernetes-cluster-architecture.svg)

Kubernetes делится на **Control Plane** (master-нода, управляет кластером) и **Worker Nodes** (выполняют рабочие нагрузки).

### Control Plane

**kube-apiserver**
- Точка входа в кластер, REST API поверх HTTP.
- Единственный компонент, который общается с etcd.
- Все остальные компоненты (kubectl, controller-manager, scheduler, kubelet) ходят через него.

**etcd**
- Распределённая key-value БД, хранит всё состояние кластера (манифесты, конфиги, статусы).
- Версии: v2 (HTTP), v3 (gRPC) — k8s использует v3.
- Постоянно делает fsync на диск → требует быстрых дисков (SSD/NVMe).

**kube-scheduler**
- Решает, **на какую ноду** поставить новый Pod.
- Учитывает при выборе:
  - **Requests/Limits** — запрошенные ресурсы (CPU/RAM) из манифеста.
  - **QoS class** — присваивается автоматически на основе requests/limits:
    - `Guaranteed` — requests == limits для всех контейнеров.
    - `Burstable` — есть requests, но не равны limits.
    - `BestEffort` — нет ни requests, ни limits (убивается первым при нехватке памяти).
  - **Affinity / Anti-affinity** — правила «размести рядом / не размещай рядом» (например, не ставить две реплики на одну ноду).
  - **NodeSelector / Taints & Tolerations** — привязка к конкретным нодам.
  - **PriorityClass** — приоритет подов; при нехватке ресурсов вытесняются поды с низким приоритетом.

**kube-controller-manager**
- Набор контроллеров, каждый отвечает за приведение реального состояния к желаемому (DeploymentController, ReplicaSetController, NodeController и т.д.).
- Работает в режиме **Reconciliation Loop** — бесконечный цикл «сравнил → исправил».

### Worker Node

**kubelet**
- Агент на каждой ноде.
- Получает спецификации подов от apiserver и говорит container runtime (containerd/Docker) их запустить.
- Следит за состоянием контейнеров, гоняет пробы, репортит статусы наверх.

**kube-proxy**
- Настраивает сетевые правила (iptables/IPVS) на ноде.
- Обеспечивает работу Service: маршрутизирует трафик с виртуального IP сервиса на реальные поды.

**Container Runtime** (containerd, CRI-O)
- Непосредственно запускает контейнеры.

## 2. Основные абстракции

### Pod
- Минимальная единица в k8s — обёртка над одним или несколькими контейнерами.
- Контейнеры внутри пода делят сеть (один IP, localhost) и тома.
- **Эфемерен**: при перезапуске получает новый IP.
- **Жизненный цикл (статусы)**:
  - `Pending` — под создан, но контейнеры ещё не запущены (тянется образ, ждём scheduling).
  - `Running` — привязан к ноде, как минимум один контейнер работает.
  - `Succeeded` — все контейнеры завершились с кодом 0 (для Job).
  - `Failed` — хотя бы один контейнер завершился с ошибкой.
  - `Unknown` — apiserver потерял связь с нодой (обычно сеть).

### ReplicaSet
- Гарантирует, что в кластере работает **N одинаковых подов**.
- Если под упал — создаёт новый.
- **Не умеет обновлять версии** — это делает Deployment.
- Параметр `replicas` в манифесте.
- **Зачем нужен**: отказоустойчивость и горизонтальное масштабирование (несколько одинаковых инстансов).

### Deployment
- Управляет ReplicaSet'ами.
- Отвечает за **обновления, откаты, историю версий**.
- При новом релизе создаёт новый ReplicaSet (v2), плавно гасит старый (v1) согласно стратегии:
  - `RollingUpdate` (по умолчанию) — постепенная замена.
  - `Recreate` — убить все старые, поднять новые.
- Откат: `kubectl rollout undo deployment/...` — переключается на старый ReplicaSet.

### Service
- Решает проблему эфемерности подов: даёт **стабильный виртуальный IP и DNS-имя**.
- Маршрутизирует трафик на поды по label-селектору.
- Балансировка по умолчанию — **round-robin** (через iptables/IPVS на kube-proxy).
- Типы:
  - `ClusterIP` — доступ только внутри кластера (по умолчанию).
  - `NodePort` — открывает порт на каждой ноде.
  - `LoadBalancer` — заказывает внешний балансировщик у облака.
  - `ExternalName` — алиас на внешнее DNS-имя.

### Ingress
- L7-маршрутизация HTTP/HTTPS трафика.
- Один внешний IP → разные сервисы по host/path (`api.example.com → service1`, `/admin → service2`).
- Требует Ingress Controller (nginx, traefik) — он реализует балансировку.
- Минус: нет единого стандарта между реализациями.

### Gateway API (новый, замена Ingress)
- `GatewayClass` — тип реализации (nginx, envoy и т.д.).
- `Gateway` — сам балансировщик (точка входа).
- `HTTPRoute` — правила маршрутизации (более гибкие, чем у Ingress).
- Решает проблемы портируемости и расширяемости.

### ConfigMap
- Объект для хранения **конфигурации** (env-переменные, файлы конфигов) отдельно от образа.
- Подключается в под как env-переменные или как файл через volume.
- **Как изменить**: `kubectl edit configmap <name>` или `kubectl apply -f cm.yaml`.
- ⚠️ Если ConfigMap подключён как env — поды **не подхватят изменения автоматически**, нужно перезапустить (например, `kubectl rollout restart deployment/...`). Если как volume — обновится сам через какое-то время.

### Secret
- То же, что ConfigMap, но для чувствительных данных (пароли, токены). Хранится в base64.

### Namespace
- Логическое разделение кластера на «папки».
- Изолирует ресурсы (Service, ConfigMap и т.д.), позволяет ставить квоты и RBAC.
- Системные: `kube-system`, `default`.

## 3. Probes (пробы) — подробно

Пробы — это **проверки kubelet'ом состояния контейнера**. Kubelet периодически дёргает их и на основе результата принимает решения.

### Виды проб

**1. Liveness Probe — «жив ли контейнер?»**
- Если падает → kubelet **перезапускает контейнер**.
- Нужна для отлова зависаний (приложение работает, но не отвечает — дедлок, утечка и т.п.).
- Пример: приложение зависло → liveness возвращает ошибку → под пересоздаётся.

**2. Readiness Probe — «готов ли принимать трафик?»**
- Если падает → под **удаляется из endpoints Service** (трафик не идёт), но **не перезапускается**.
- Нужна, когда приложение запущено, но временно не готово: прогревает кэш, переподключается к БД, мигрирует и т.д.
- Как только снова отвечает — возвращается в балансировку.

**3. Startup Probe — «приложение стартануло?»**
- Выполняется **только при старте**, до первого успеха.
- Пока не пройдёт — liveness и readiness **не запускаются**.
- Нужна для медленно стартующих приложений (Java, инициализация БД), чтобы их не убил liveness раньше времени.

### Типы проверок
- `httpGet` — HTTP-запрос, проверка кода ответа (200–399 = OK).
- `tcpSocket` — попытка открыть TCP-соединение.
- `exec` — выполнить команду в контейнере (exit 0 = OK).
- `grpc` — gRPC health check.

### Параметры
- `initialDelaySeconds` — сколько ждать после старта контейнера перед первой проверкой.
- `periodSeconds` — интервал между проверками.
- `timeoutSeconds` — таймаут одной проверки.
- `failureThreshold` — сколько подряд неудач, чтобы признать проверку проваленной.
- `successThreshold` — сколько подряд успехов, чтобы вернуться в «здоровое» состояние (после серии неудач). Сбрасывает счётчик неудач.

**Пример логики successThreshold = 2**: было 0,0,0,0,1 → ещё одного успеха не хватает, состояние всё ещё «нездоровое». После 0,0,0,1,1 → счётчик неудач обнуляется.

## 4. Автоскейлинг

**HPA (Horizontal Pod Autoscaler)**
- Меняет количество подов (replicas) в Deployment'е.
- Метрики: CPU, RAM, кастомные (через metrics-server).
- Принцип: каждые ~15 сек смотрит метрики → если средняя нагрузка выше target → увеличивает replicas, ниже → уменьшает.
- Формула: `desiredReplicas = ceil(currentReplicas * (currentMetric / targetMetric))`.

**VPA (Vertical Pod Autoscaler)**
- Меняет requests/limits самого пода.

**Cluster Autoscaler**
- Добавляет/удаляет **ноды** в кластере, когда подам не хватает места.

## 5. Полезные команды (для собеса)

```bash
# Логи пода
kubectl logs <pod-name>
kubectl logs <pod-name> -c <container>          # конкретный контейнер
kubectl logs <pod-name> -f                       # follow (как tail -f)
kubectl logs <pod-name> --previous               # логи предыдущего упавшего контейнера

# Изменить ConfigMap
kubectl edit configmap <name>
kubectl apply -f configmap.yaml
# Чтобы поды подхватили (если cm в env):
kubectl rollout restart deployment/<name>

# Применение манифестов
kubectl apply -f file.yaml      # идемпотентно (рекомендуется)
kubectl create -f file.yaml     # не идемпотентно, упадёт если есть

# Диагностика
kubectl describe pod <pod>      # события, статусы, причины
kubectl get pods -o wide
kubectl exec -it <pod> -- sh
```

## 6. Reconciliation Loop (ключевая идея k8s)

Всё в k8s работает по принципу:
1. Пользователь описывает **желаемое состояние** в манифесте (declarative).
2. Контроллеры в бесконечном цикле сравнивают желаемое и реальное состояние.
3. Если есть расхождение — приводят к желаемому.

Поэтому если убить под вручную — Deployment поднимет новый. Если изменить replicas — ReplicaSet создаст/удалит поды.
