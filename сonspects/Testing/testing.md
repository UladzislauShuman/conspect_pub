## Уровни

*список нужно читать снизу вверх, как пирамиду. чем выше, тем дороже. чем выше, тем медленнее выполняются

- E2E
    - межсистемное взаимодействие
    - для глобального регресса
    - сложные и дорогие
    - как правило -- dev и поддерживают qa
- Integration
    - проверяют взаимосвязь компонентов кода
    - проверяют функционал API
    - зависимости на внешние системы, как правило, не проверяются
    - dev и qa
- Unit
    - изолированные
    - легковесные
    - проверяют небольшой элемент функционала
    - пишуться разработчиком

## Ручное и Авто-тестирование

когда автотесты круто (если)
- руками регрессионное проверять -- несколько недель
- долгосрочный и масштабный проект
- частые релизи и часто требуется прогонять тест-кейсы
- при нагрузочном тестировании

когда все-таки ручное
- краткосрочный проект
- разработка MVP
- UI и UX тестирование
- исследовательское тестирование

## Показатели качества кода

не являются показательными
- количество багов
- процент покрытия тестами
- процент охвата модульного тестирования

являются показательными
- процент критичных багов
- время тестирования
- время устранения багов
- процент отклоненных багов

откуда беруться отклоненные баги
- неполная или неточная спецификация 
- несогласованы тест-кейсы

вариант для решения этой проблемы -- 3 амиго (SA, DEV, QA)

модели зрелости
- понимание текущего уровня команды
- выделение областей для развития и проработка шагов совершенствования процессов

уровни
- программное обеспечение должно работать правильно
    - нет областей процесса поставки
    - нет / минимально использование автотестов
    - цель тестирование -- что ПО работает
    - нет гарантий качества поставки
- соотвествует спецификации
    - тестирование как отдельный этап поставки
    - базовые методики тестирования
    - цель -- првоерить, что ПО работает по спецификации
- тестирование интегрированно в ЖЦ ПО
    - цели тестирования формируются на требованиях
    - автоматизация процессов тестирования
- тестирование измеримо
    - количественно можно определить
    - баги регистируются
    - уровни критичности дефектов определены
- тестирование оптимизируемо
    - контроль эффективности и затрат на тестирование
    - практикуется контроль качества и профилактика дефектов
    - процесс тестирования можно настраивать и совершенствовать

## Непрерывное тестироване
- ускоряет процесс поставки
- повышает качество засчет многоэтапного тестирования
- сокращает затраты

## Ошибки при тестировании

- только проверка happy path
- все за один раз 
- внешние зависимости (особенно в Unit-тестах)
- опираются только на конкретную реализацию
- изменять код сервиса для задач тестирования
- нарушение One assert rule

## Нагрузочное

- для оценки лимита производительности
- поиск проблемных мест 
- еще
    - среднее время ответа сервиса
    - пиковое время ответа сервиса
    - рост ошибок

когда нагрузочный тест Валиден
- сценарий должен включать полноценный флоу поведения пользователя
