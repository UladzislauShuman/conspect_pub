## База

### REpresentational State Transfer

это архитектурный стиль для создания веб-сервисов

что дает
- масабируемость системы
- общность интерфеса
- независимо внедрять
- промежуточные компонты, что снижают задержку, увеличивают безопасность

когда
- есть ограничение пропускной способности
- нужно кешировать запросы
- необходимо масштабировать систему
- AJAX (подход к созданию интерактивных сайтов, где данные загружаются в фоновом режиме. Страница обновляется по частям, без полной перезагрузки)

преимущества
- нет доп внутренних прослоек
- каждая единица инфы однозначно определяется по URL
- управление инфой зависит От протокала передачи данных(например -- HTTP и его методы GET, PUT, POST, DELETE)

### RESTful

критерии такого приложения
- Client-Server
- Stateless -- все что нужно в запросе
- Cache -- ответ должен быть отмечен -- кэшировать или нет
- Uniform Interface -- единый интерфейс между клиентом и сервером
    - принципы
        - идентификация ресурсов -- является ресурсом все то, чему можно дать имя
        - манипуляция над ресурсами через представления 
        - само-документируемые сообщения -- запрос и ответ должны хранить в себе всю необходимую информацию для их обработки. дополнительного быть не должно
        - hypermedia as the engine of application state
- Layered System -- можно делить на слои. каждый слой знает только о следующем. не предыдущем и не следуюЩИХ
- Code on demand -- необязательно. можно передавать код, который будет выполняться на стороне клиента

Не связана ни с какой технологией, но тесно ассоциируется с HTTP

### Индемпотентность

когда кучу раз одного и тоже -- одинаковый результат

но -- важно что -- состояние не меняется, но при этом ответ может меняться (DELETE -- сначала 204, потом 404 -- состояние не поменялось, но при этом ответ разный)

- PUT и DELETE по определению идемпотентны
- GET, HEAD, OPTIONS и TRACE определены как безопасные -- только для получения информации и не должны изменять состояние сервера

Безопасная => идемпотентная 

### HTTP методы и RESTful

- GET -- получение данных
    - 200
    - 400, 404
    - идемпотентен
- PUT -- обновление данных (полностью или частично)
    - не безопасный. ибо модифицируется ресурс
    - 200, 201, 204
- POST -- создание нового ресурса. вложенных тоже
    - 201 + адрес нового ресурса
- DELETE -- удаление ресурса по URI
    - 200, потом уже 204, может даже 404
    - идемпотентен (но если мы -- декрементируете счетчик -- то уже нет)

#### Немного цифр

- 1xx: Information
    - 100: Continue
- 2xx: Success
    - 200: OK
    - 201: Created
    - 202: Accepted
    - 204: No Content
- 3xx: Redirect
    - 301: Moved Permanently
    - 307: Temporary Redirect
- 4xx: Client Error
    - 400: Bad Request
    - 401: Unauthorized
    - 403: Forbidden
    - 404: Not Found
- 5xx: Server Error
    - 500: Internal Server Error
    - 501: Not Implemented
    - 502: Bad Gateway
    - 503: Service Unavailable
    - 504: Gateway Timeout

## Вопросы

### `что REST отличает от обычного общения по HTTP`

`HTTP` -- это транспортный протокол, а `REST` -- это набор правил по его использованию

вероятно речь идет об `RPC` -- Remote Producer Call как о Обычном `HTTP` общении

- в `RPC` мы обращаемся к функциям (*GET /deleteUser?id=5*) , а в `REST` к ресурсам (*DELETE /users/5*)(глагол это метод а `URL` -- существительное)

- в `HTTP` обычно для всех операций используют только `GET` и `POST`, а в `REST` -- строго весь сперкор методов согласно их назначению

- `HTTP` часто использует сессии, а `REST` -- Stateless

- в обычном `HTTP` ответ может быть любым -- хоть HTML, хоть строка текста, редирект. А `REST` стремиться к передаче состояния ресурса (JSON, XML)

- `HATEOS`. в `HTTP` клиент должен знать заранее все `URL` системы. а `REST` -- сервер в ответе присылает не только данные, но и ссылки на то, что с ним можно сделать дальше (данные о заказе. ссылку на отмену или оплату)

---

### `можно ли менять местами GET и POST`

да, поменять можно -- технически, но архитектурно и согласно стандартам -- так нельзя делать, ибо 
- `GET` должен быть `Safety` -- не изменять состояния сервера
- `GET` `идемпотентен`, а `POST` по своей природе нет. если мы заменим `POST` на `GET`, то браузер или прокси-сервера могут решить, что запрос повторять не нужно
- результаты `GET` кешируются браузерами, прокси-серверами, CDN. из-за чего -- если мы выполняем транзакцию, то может достать из кэша и написать "успешно", хотя на самом деле деньги не списывались

- GET передает данные в URL(`query параметры`). длина URL До 2048 симвлов. и данные видны в логах сервера и истории браузера. а `POST` передает данные в `Body`. объем практичекси не ограничен, и они скрыты от глаз (но без `HTTPS` они все равно не зашифрованы)

### `что вы знаете о Cookie и что с ними можно делать`

`Cookie` -- небольшие фрагменты текстовых данных, которые сервер отправляет браузеру. браузер сохраняет их у себя и автоматически прикрепляет к каждому последующему зарпосу этому же серверу

`как работает`
- сервер -> клиент: в HTTP ответе сервер передает заголовок Set-Cookkie с данными
- клиент -> сеорвер: при следующем запросе браузер сам добавляет заголовок

`что с ним можно делать `
- `управление сеансами` -- хранить ID сессии, корзины покупок
- `песонализация` -- хранение настроек пользователя 
- `трекинг и аналитика`

`ключевые атрибуты (технические настройки)`
- `HttpOnly` -- запрет к cookie через JS. как защита от XSS-атак
- `Secure` -- указывает браузеру отправлять cookie только по HTTP
- `SameSite` (Strict, Lax, None) -- защищает от `CSRF-атак`. определяет -- будет ли `cookie` отправляться при переходах со сторонних сайтов
- `Expires/ Max-Age` -- время жизни `cookie`. по умолчанию -- сессионая (при закрытии браузера закрывается)
- `Domain / Path` -- ограничиывает область видимости cookie конкретным доменом или путем на сервер

`ограничения`
- `размер` до 4кб на штуку
- `количество` около 20-50 штук на один домен (от браузера зависит)
- `производительность` -- cookie пересылают при каждом запросе, и если их будет избыточно много, то это может замедлить сеть

`типы`
- `необходимые` -- чтобы сайт корректно работал
- `функциональные` -- для обеспечения индивидуального опыта использования сайта 
- `статистические` -- чтобы, например, хранить историю посещений страниц сайта

`еще какие-то типы по какому-то признаку`
- сессионные
- постоянные
- сторонние
- супер
- зомби

### `какие еще сценарии хранения данных у пользователя вы знаете`

- `Web Storage API` -- самые популярные альтернативы cookie. хранение пар Ключ-Значение
    - `LocalStorage`
        - жизнь: `вечно` (даже при закрытии вкладки, браузера, перезагрузке компа)
        - объем: `5-10 мб` (браузер)
        - сценарий: настройки темы, id корзины -- которые должны быть доступны `при следующем визите`
    - `SessionStorage`
        - жизнь: пока открыта `конкретная вкладка`. если этот же сайт в другой вкладке открыть, то там будет свой `SessionStorage`
        - объем: как `LocalStorage`
        - `сценарий`: состояние многошаговой формы (чтобы при случайной перезагрузке страницы данные не пропали), временные фильтры поиска

- `IndexedDB` -- NoSQL бд в браузере
    - как работает -- транзакционная, ОО-ешная, асинхронная. можно хранить даже сложные объекты, файлы
    - объем -- зависит `от объема ЖД` пользователя
    - сценарий -- создание `Progressive Web Apps`, работа прилаг в `offline` (google docs или почтовые клиенты, которые хранят данные локально)

- `Cache API`
    - как работает -- позволяет хранить пары `"Запрос - ответ"`
    - сценарий -- кэширование `статических ресурсов`, чтобы мгновенно загружалось без интернета

- `URL` (`Quert Parameters` и `Hash`)
    - `сценарий`: фильтры, сортировки, номер страниц в списке
    - плюсы -- можно `поделиться` легко с другом

- `Memory (JS Variables)`
    - `до первой перезагрузки` страницы (F5). самый быстрый доступ, но недолговечна
