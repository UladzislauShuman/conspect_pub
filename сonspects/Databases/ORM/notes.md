## Ответы

### Что такое проблема N+1 и как её решить?

это ситуация в ORM, когда для загрузки коллекции объектов из БД выполняется один начальный SQL запрос, а затем для загрузки связанных данных каждого из этих объектов выполняется еще N (количество объектов) дополнительных SQL запросов

```java
List<Author> authors = entityManager.createQuery("SELECT a FROM Author a", Author.class).getResultList();
// 1 запрос: SELECT * FROM Author; (Допустим, вернулось 10 авторов)

for (Author author : authors) {
    // При обращении к геттеру Hibernate понимает, что книги не загружены, и идет в БД
    System.out.println(author.getBooks().size()); 
    // Еще 10 запросов (по одному на каждую итерацию цикла): 
    // SELECT * FROM Book WHERE author_id = ?
}
```

способы решения
- JOIN FETCH -- одним запросом говорим, чтобы он подтянул связанные сущности
```java
List<Author> authors = entityManager.createQuery(
    "SELECT a FROM Author a JOIN FETCH a.books", Author.class).getResultList();

for (Author author : authors) {
    System.out.println(author.getBooks().size());
}
```

- @EntityGraph в Spring Data JPA -- просто как альтернатива, вместо запросов как в предыдущем примере
```java
@EntityGraph(attributePaths = {"books"})
List<Author> findAll();
```
- @BatchSize -- загружаем пачками
```java
@OneToMany(mappedBy = "author")
@BatchSize(size = 10) // Настраивается над коллекцией
private List<Book> books;
// если 100 авторов, то 1 запрос на авторов + 100 / 10 чтобы книги собрать 
```

- FetchMode.SUBSELECT (для Hibernate) -- сначала всех авторов, а потом,При первом вызове, сложным SELECT запросом, достанет Все книги


### В чём разница между Lazy и Eager загрузкой?

основная разница -- когда Именно происходит запрос на допулучение связанных данных из БД

- Eager Loading (Жадная, Ранняя загрузка)
    - FetchType.EAGER
    - когда запрашивается основная сущность, то сразу делается запрос и на получение всех связанных с ней. и не важно -- воспользуюсь ими или нет
- Lazy Loading
    - FetchType.LAZY
    - когда основную часть достали, связанная не достается. вместо них подставляется Proxy-объект. реальный запрос выполнится только при первом обращении

Какие ставятся по умолчанию
- @OneToMany -- LAZY -- чтобы не вытащить миллион записей
- @ManyToMany -- LAZY
- @ManyToOne -- EAGER
- @OneToOne -- EAGER

### Когда использовать Lazy, а когда Eager loading?

Главное правили Gemini: всегда и везде использовать LAZY, EAGER не должен использоваться практически никогда

даже для @ManyToOne и @OneToOne менять

почему это антипаттерн?
- проблема производительности и "декартово произведение"
    - если у сущности много связей, то при findAll hiber сгенерит огромный заропс с кучей OUTER JOIN. мы достанем тонну данных, хотя нужен был только orderId
- проблема потери контроля
    - EAGER -- мы не сможем отменить эту жадную загрузку
    - LAZY -- позволяет решать на уровне конкретного запроса, надо оно нам Сразу или можно Лениво и позже
- сложность с N + 1
    - Eager тоже может быть причиной N+1. тоже -- 1 запрос + N на подтягивание Eager полей

когда Можно использовать?
- связь 1 к 1, которая нужна Всегда
- глобальные справочники -- если вы загружаете какую-то конфигурационную сущность при старте приложения один раз, и она тянет за собой пару мелких справочников

### Что такое LazyInitializationException и как её избежать?

выбрасывается, когда мы пытаемся обратиться к ленивой коллекции или связаной сущности (через getter) в тот момент, когда сессия с Hibernate уже закрыта

пример flow
- открываем транзакцию (метод в классе @Service)
- загружаем User, но его roles ленивые, поэтому туда Hibernate положил proxy-объект
- транзакция завершается (метод завершается), транзакция закрывается
- в слое контроллера мы хотим получить роли
- proxy объект хочет пойти в базу, но видит что сессии нет
- LazyInitializationException

как избежать 
- JOIN FETCH -- все те данные, которые нужны нам точно понадобятся, были загружены сразу
```java
@Query("SELECT u FROM User u JOIN FETCH u.roles WHERE u.id = :id")
User findUserWithRoles(@Param("id") Long id);
```

- @Transactional на уровне контроллера 
    - вроде как она по умолчанию в Spring, но проблема -- транзакция висит даже то время, пока мы формируем JSON или HTML. лучше сделать spring.jpa.open-in-view=false

- Hibernate.initialize() указываем явно, чтобы инициализировал proxy
```java
@Transactional
public User getUser(Long id) {
    User user = repo.findById(id);
    Hibernate.initialize(user.getRoles()); // Принудительная загрузка
    return user;
}
```

- DTO
    - сущность, не из базы. мы в сервисе из сущности БД маппим в простой Java класс (dto), что содержит только нужные поля. в процессе маппинга данные и подтянутся. и мы получаем объект, который уже никак не связан с БД


### Какие стратегии fetch существуют в Hibernate?

- про момент загрузки. FetchType
    - LAZY -- по требованию
    - EAGER -- немедленная
- по способу формирования SQL запроса. FetchMode
    - SELECT -- выполняет отдельный SELECT запрос для каждой связи
        - загрузил объект. когда нужна коллекция в нем -- еще один запрос
        - позволяет работать ленивой загрузке
        - риск N + 1
    - JOIN -- загрузить все одним запросом, используя OUTER JOIN
        - сразу генерирует запрос с JOIN
        - ВАЖНО: если использовать FetchMode.JOIN, то FetchType.LAZY игнорируется и все равно используется EAGER
        - проблема: если много JOIN, то запрос тяжелеет, а результат в памяти выглядит как "декартово произведение" (дублирование строк), а это нагружает RAM
    - SUBSELECT -- самая умная стратегия для коллекций
        - Hibernate делает два запроса
        - первый достает список родителей (авторов)
        - второй достает все книги для всех этих авторов сразу
        - Плюс: решает N+1 и без тяжелых JOIN

### Что делает аннотация @BatchSize?

как один из способов решения проблемы N+1

Вместо того, чтобы делать отдельный Select для каждого объекта, Hibernate собирает ID нескольких объектов и делает один запрос с оператором IN

#### Пример 
- без @BatchSize -- отдельный запрос `SELECT * FROM Book WHERE author_id = 2` ...
- с -- `SELECT * FROM Book WHERE author_id IN (1, 2, 3, 4, 5)`, когда дошли до 6-ого -- `SELECT * FROM Book WHERE author_id IN (6, 7, 8, 9, 10)`

Где можно ставить аннотацию?
- над коллекцией, которая поле класса
- нам самим классом 

плюсы
- универсальность -- работает автоматически при любом доступе к ленивым полям
- просто -- не нужно переписывать JPQL запросы или использовать EntityGraph
- снижение N+1 -- не исчезает полностью, но уменьшается в size раз

минусы
- все равно есть лишний запрос
- статичность -- размер батча защит в код и его нельзя поменять в зависимости от ситуации

### Опишите жизненный цикл Entity в Hibernate

выделяют 4 состояния
- New (Transient) 
    - обычный Java-объект, который только что создали с помощью new 
    - связь с БД: ее нет (нет еще такой записи)
    - связь с сессией: тоже нет (Hibernate про него ничего не знает)
    - действие: если поменять поле в объекте, то в базе ничего не поменяется

- Managed (Persistent) -- Управляемы
    - объект привязан к активной сессии и представляет запись в БД
    - связь с БД: есть (строка в таблице)
    - связь с сессией: есть 
    - особенность: в этом состоянии работает механизм Dirty Checking (Hibernate автоматически отслеживает изменения полей объекта и при закрытии транзакции (или вызове flush) сам выполнит нужный UPDATE в базе. нам не нужно вызывать save вручную)
    - как попасть в это состояние: вызвать persist(), merge() на Transient объекте или просот достать объект из базы через find() или запрос

- Detached -- отсоединенный 
    - быль ранее управляемый, но сейчас связь с сессией разорвана
    - связь с БД: есть -- запись есть в БД
    - связь с сессией: нет (сессия закрыта или объект был явно удален из нее через evict())
    - особенности: изменения в объекте больше не отслеживаются
    - как попасть: закрыть транзакцию или сессию Или вызвать clear()

- Removed -- удаленный
    - все еще привязан к сессии, но помечен как удаленный
    - связь с БД: пока есть, но будет удалена после flush() (в SQL выполняется DELETE)
    - связь с сессией: есть
    - как попасть: метод remove()

#### Схема переходов (Transitions)

1.  **Transient -> Managed:** `persist()`, `save()`.
2.  **Managed -> Detached:** `close()`, `clear()`, `evict()`.
3.  **Detached -> Managed:** `merge()`.
4.  **Managed -> Removed:** `remove()`.
5.  **Removed -> Managed:** `persist()` (отмена удаления до коммита).
