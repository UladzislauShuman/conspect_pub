### Null-safty
- проблема 
    - проблема NullPointerException в Java состоит в том, что -- компилятор позволяет тебе вызвать метод у объекта, который может быть null. причем ошибка будет только в Runtime
    - из-за этого, разработчикам приходиться писать кучу условий `if (object != null) { ... }`
- решение
    - Возможность явно указать, что переменная может быть null (Nullablity). и проверка происходит на этапе компиляции
    - Например: `String?` и `String`
        - компилятор будет проверять, чтобы ты не мог вставить Null пременную в NoNull переменную
        - ты не сможешь вызывать метод у `String?`, если не воспользуешься `?`
        - например, `str?.length()` 
            - если str == null, то вернется null, иначе вернется длина строки
    - примеры
        ```kotlin
        var b: String? = null
        val length: Int? = b?.length // length == null
        val length: Int = b?.length ?: 0 // length == 0

        if (b != null) {
            // теперь b не String?, а String
        }
        ```
- из этой же оперы
    - решает не "искоренить NPE", а убрать неожиданный null. при этом нам важно оставить обратную совместимость с Java
        - по поводу обратной совместимости, есть понятие `Platform Types`. они позволяют использовать Java код в Kotlin коде, но не позволяют использовать Kotlin код в Java коде. Обозначаются `String!`
    - `?:` -- это Elvis оператор. он возвращает значение, если оно не null, иначе возвращает значение справа от оператора
        ```kotlin
        var b: String? = null
        val length: Int = b?.length ?: 0 // length == 0
        ``` 
    - `?.` -- это safe call оператор. он возвращает null, если значение справа от оператора равно null
    - `!!` -- это non-null assertion оператор. он возвращает значение, если оно не null, иначе выбрасывает NPE
        - считается плохим тоном его использовать

- как это работает под капотом
    - для JVM `String?` и `String` это одно и тоже 
    - компилятор Kotlin добавляет 
        - `@Nullable` и `@NotNull` 
        - в начале каждого метода, что принимает non-nullable параметры, компилятор вставляет `Intrinsic.checkNotNullParameter(param, "paramName)`. В случае если, выкенет `IllegalArgumentException`

- какие есть еще каверзные вопросы
    - почему Smart Cast иногда не работает?
        - (это когда мы делаем проверку на `!= null` и далее наша `String?` уже ведет себя как `String`)
        - если `val x: String?`, то оно сработает
        - если `var x: String?`, то оно не сработает
        - ибо "в многопоточной среде `var x` могло измениться и стать null
    - Предположим у меня поздняя инъекция зависимостей в Spring. Что делать?
        - если предложить сделать nullable `?`, то он будет писать много `?.`
        - есть `lateinit var` -- модификатор, который говорит, что "я обязательно сюда что-то До того, как начну из него читать"
            - если ты забудешь, то вылезит исключение `UninitializedPropertyAccessException`
            - работает только с `var` и не работает с примитивами

    - ситуация```kotlin
    class Box<T>(val item: T)``` 
    может ли item быть null?
        - да, может. ибо неявно расширяет `Any?`
        - чтобы это исправить, нужно ```kotlin
        class Box<T : Any>(val item: T)```

    - String как бы является под типом String?

- пример ```kotlin
    val str: String? = null
    println(str.isNullOrEmpty())```
    - это функция расширения для нашего nullable типа
    - под каппотом, в Java, этот метод превращается в статический метод, куда мы можем передать null как аргумент

### data класс
- проблема
    - чтобы создать в Java обычный DTO класс, это нужно Очень много кода и реализаций делать самому (без @Lombok и record-ов)
- решение
    - добавили data классы
- что дает
    - предположим ```
    data class User(val id: Int, val name: String)```
    - тогда сгенерируется 
        - `equals()` -- сравнение по значенияем полей
        - `hashCode()` -- для hash-СД
        - `toString()`
        - `copy()` -- копирование объекта. при этом в процессе можно поменять поле ```user.copy(name = "Bob")
        - componentN() -- для деструктуризации ```val (id, name) = user```
- немного Bytecode
    - по сути -- просто генерит новый код
    - с появлением record в java можно использовать аннотацию @JvmRecord

- Trap
    ```
    data class Person(val name: String) {
        var age: Int = 0
    }
    val p1 = Person("Ivan").apply { age = 20 }
    val p2 = Person("Ivan").apply { age = 30 }
    p1 == p2 // true
    ```
    - ибо все эти методы генерируются только для полей из первичного конструктора
    - data-класс может наследоваться от других, но от него наследоваться -- нет
    - copy() делает поверхностное копирование (копирует не поля, а ссылки на них)
- пример
    ```kotlin
    data class Image(val pixels: ByteArray)

    val img1 = Image(byteArrayOf(1, 2, 3))
    val img2 = Image(byteArrayof(1, 2, 3))

    println(img1 == img2) 
    ```
    false, ибо -- equals() использует ==, а массивы по ссылкам сравниваются, а не по значениям в них. как решение -- переопределить их вручную. в List кстати все работает из коробки   
- еще вопрос
    - может ли data class быть abstract, open, sealed или inner?
    - нет. он всегда final. был бы он open, то поломался бы equals()
### sealed класс

позволяет ограничить набор типов (подклассов)

что позволяет делать 
- например у нас бекенд, делаем запрос куда-то и у нас есть несколько типов ответов -- положительный, отрицательный и тп
мы могли бы сделать какой-то общий класс, в котором бы было кучу пустых полей, но мы можем сделать иначе
```kotlin
sealed class Result {
    data class Success(val data: String) : Result()
    data class Error(val code: Int, val message: String) : Result()
    object Loading : Result()
}
```
- есть классная вещь -- исчерпывающий when (Exhaustive when) -- так как компилятор знает все возможнные подклассы Result (их не получиться создать где-то в другом модуле или библиотеке), он заставит нас обработать их все. \
причем ветка else не будет нужна, ибо мы и так все варианты обработали. \
это хорошо тем, что -- кто-то добавит новый подкласс, и компилятор скажет что "его нужно обработать"

- немного Bytecode
    - до jvm17 -- какой-то абстрактный класс с приватным конструктором
    - после -- использовал уже то, что завезла JVM

- вопросы 
    - могу ли я создать подклассы в другом месте? -- нет. должны быть объявлены в том же пакете и модуле (ранее было и в одном файле). надо чтобы компилятор  мог точно собрать их на этапе компиляции
    - зачем добавили sealed interface?
        - в котлин нет множественного наследования. можно интерфейсы. можно несколько интерфейсов. вроде как добавили в котлин 1.5, где можно в несколько модулей в одном пакете, а не в одном файле. ну и тем самым стало удобно и возможно
    - зачем sealed, если есть enum class? в чем концептуальное отличие
        - enum -- это ограничение значений. каждый его элемент -- имеет единственный экземпляр. не может быть разного состояния 
        - sealed class -- это ограничение типов. я могу сделать кучу экземпляров одного такого класса, и он будет разный внутри
    - о базовом sealed class-е
        - можно ли добавить общее поле в базовый класс?
        - можно 
        ```kotlin
        sealed class NetworkCall(val timestamp: Long) {
            data class Success(val data:  String, val time: Long) : NetworkCall(time)
            data class Error(val code: Int, val time: Long) : NetworkCall(time)
        }
        ```
        - а можно ли в interface?
        - нет. можно объявить val что-то, но каждый наследник будет хранить его сам

### Destruction (деструктуризация)

это сахар, который позволяет распаковать объект на несколько переменных за одну строчку

```kotlin
val user = User(1, "Ivan")
val (id, name) = user
```

как это работает?

компилятор превращает эту строку в вызов спец функций component<number>()

```kotlin
val id = user.component1()
val name = user.component2()
```

у data-классов они генеряться автоматически, но я могу их и сам оперделить

```kotlin 
class Point(val x: Int, val y: Int) {
    operator fun component1() = x
    operator fun component2() = y
}

val (x, y) = Point(x, y)
```

так же ее классно в циклах использовать

важно кстати помнить

```kotlin
data class User(val id: Int, val name: String, val age: Int) 

val user = User(1, "Vlad", 19)
val (id, _, age) = user
```

## Категория 3: Функции и Магия Kotlin (Scope functions, Extensions, DSL)

### Extension-функции

- что это
    - механизм, который позволяет "добавлять" новый метод или свойство к уже имеющемуся классу, не наследуясь от него и не меняя его исходный код

    ```kotlin
    fun String.removeFirstAndLast(): String {
        return this.substring(1, this.length - 1)
    }

    val myStr = "Hello"
    print(myStr.removeFirstAndLast())
    ```

- зачем?
    - чтобы расширять классы, к исходному коду которых у меня нет доступа (стандартная библиотека Java, Spring и тп). тем самым сделать более читаемым

- плюсы и минусы
    - плюсы
        - читаемость
        - автокомплит
    - минусы    
        - но можно и засорить автокомплит лишним
        - нельзя работать с private или protected. только public

- когда это 
    - реально Тема!
        - маппинг
        - форматирование данных
    - а когда не очень
        - помещать большую Бизнес Логику. это не замена сервисного слоя

- ловушки
    - как это в Java выглядит?
        - в статический метод, где сам объект передается первым аргументом
        - класс такой примерно получается
            - если у меня был модуль StringExt.kt, где был метод
            - то создается StringExtKt, который уже содержит наш метод
            - и да, этот класс можно вызвать из Java
            - так же его можно переименовать @JvmName("Name")
    
    - а что с полиморфизмом? -- его нет
    ```kotlin
    open class Parent
    class Child : Parent()

    fun Parent.printName() = println("Parent")
    fun Child.printName() = println("Child")

    fun main() {
        val obj: Parent = Child()
        obj.printName() // Parent
    }
    ```
    
    - а что с null? -- да, только если для nullable-типа
    ```kotlin
    fun String?.isNullOrEmpty(): Boolean {
        return this == null || this.isEmpty()
    }
    val s: String? = null
    println(s.isNullOrEmpty()) // true
    ```
    
    - свойтво
        - свойство -- оно вычисляемое
        - а состояние -- нельзя
        ```kotlin
        // ХОРОШО
        var String.lastChar: Char
            get() = this[length - 1]

        // ПЛОХО
        var String.customData(): String = "text"
        ```
    
    - методы для `String?` видны для `String`, но наоборот нет
        - это можно обойти 
            - Smart Cast
            - `?.`

- пример Когда реально круто 
    - вместо постоянного 
    ```kotlin
    request.getHeader("Authorization")?.removePrefix("Bearer ")
    ```
    - можно
    ```kotlin
    fun HttpServletRequest.extractJwtToken(): String? {
        return this.getHeader("Authorization")?.takeIf { 
            it.startsWith("Bearer ") 
            }?
            .substring(7)
    }
    ```

### Scope functions (let, run, with, apply, also)

- что это такое?
    - 5 стандартных функций, единственная цель которых -- выполнить блок кода/лямбду в контексте конкретного объекта. внутри этого блока я могу обращаться к объекту по this или it

- зачем?
    - сахар
    - избавиться от создания промежуточных переменных
    - работать с null

- плюсы и минусы
    - плюсы
        - fluent style -- когда метолы объекта возвращают его же (текущий или новый экземпляр)
    - минусы
        - можно сделать нечитаемым

- о каждой функции
    - apply 
        - этот как когда я покупаю новый телефон и делаю над ним настройки. и в результате я хочу получить Телефон этот же и уже настроенный
    - also
        - сделай Это , А ЕЩЕ... 
        - она берет объект, дает возможность сделать с ним какое-то действие через it (логирование, например). и возвращает сам объект
    - with 
        - с, при помощи 
        - единестваенная является тут extenction
        - и принимает объект как аргумент, дступы по this
        - пример
        ```kotlin
        val status = with(printer) {
            turnOn()
            printDocument() // статус печати
        }
        ```
    - run
        - в контексте этого объекта, запусти этот код и дай результат
        - this
    - let
        - пусть, позволить
        - "пусть этот объект -- это переменная it, тогда давай сделаем что-то с нем и вернем новый результат"
        - user?.let {...}
            - если пользователь есть, то давай я сделаю... 

- о this и it
    | Характеристика | `this` (apply, run, with) | `it` (let, also) |
    | :--- | :--- | :--- |
    | **Краткость** | ✅ Максимальная. Пишешь просто `name = "Ivan"`. | ❌ Нужно всегда писать `it.name = "Ivan"`. |
    | **Явность** | ❌ Может быть непонятно, чей метод вызывается, если их много. | ✅ Всегда понятно, что ты работаешь с объектом `it`. |
    | **Переименование** | ❌ Нельзя переименовать. Всегда `this`. | ✅ Можно дать осмысленное имя: `user ->`. |
    | **Вложенность** | ❌ Опасно. Легко запутаться в слоях `this`. | ✅ Безопасно, если давать имена переменным. |

- ловушки 
    - основное их отличие между собой
        - Как объект доступен: this или it
        - Что возвращает функция: сам объект или результат выполнения
    - 
    | Функция | Доступ к объекту | Что возвращает | Является ли Extension? |
    | :--- | :---: | :--- | :--- |
    | **`let`** | `it` | Результат лямбды | Да (`obj.let {}`) |
    | **`run`** | `this` | Результат лямбды | Да (`obj.run {}`) |
    | **`with`** | `this` | Результат лямбды | **Нет** (`with(obj) {}`) |
    | **`apply`** | `this` | Сам объект | Да (`obj.apply {}`) |
    | **`also`** | `it` | Сам объект | Да (`obj.also {}`) |
    
    - пример
    ```kotlin
    class UserProfile {
        val id = 100

        fun update(user: User) {
            user.apply {
                print(id) // если хочу внешний, то this@UserProfile.id
            }
        }
    }
    ```

### Функции высшего порядка и Top-level функции.

- что это 
    - верхнего уровня -- функиции, что написаны просто в файле .kt, вне всяких классов, интерфейсов или объектов
    - высшего порядка -- функции, что принимают другую функцию как аргумент или возвращают ее как результат 

- зачем?
    - верхнего уровня
        - чтобы не писать Классы-утилиты. ибо в Java функцию без класса сделать нельзя
    - высшего порядка 
        - чтобы передавать поведение, а не объект

- плюсы и минусы
    - плюсы
        - код гибче, красивее. меньше boilerplate кода
    - минусы
        - функции высшего порядка бьют по производительности и памяти (каждая переданная лямбда -- это новый объект в куче. если сделать цикл из 1000 элементов, то будет не очень)
        - читаемость можно сломать
        - сложность отладки (трейс становиться длиннее и запутаннее из-за вложенных вызовов и сгенерированных классов)

- примеры 
    - top level
        - глобальные хелперы
        - математические вычисления
        - константы
    - высшего порядка
        - операции для транзакций
        - ретраи
        - обработка коллекций

- вопросы
    - как выглядит в Java top-level?
        - если я создал свой Utils.kt, там функция, то создается класс UtilsKt и наш метод там уже public static
        - можно переименовать @file:JvmName("MyUtils")
    - как выглядит в Java Высшего порядка?
    ```kotlin
    fun process(action: (Int) -> String)
    process { number -> number.toString() }
    ```
    ```java
    process(new Function1<Integer, String>) {
        @Override
        public String invoke(Integer number) {
            return number.toString();
        }
    }    
    // отсюда и проблема с памятью
    // мы каждый раз создаем новый объект анонимного класса
    ```
    - еще есть когда -- внутри лямбды используешь объявленную вне переменную. тогда компилятор вынужден передать ее во внутрь анонимного класса. а это еще больше бъет по памяти. а это называется "захват переменной"

    - сравнение Top-level и companion object
        - если функция не завязана на логике конретного класса, то стоит ее сделать top-level


### Магия `inline`

"Ты сказал, что функции высшего порядка создают объекты FunctionN и нагружают Garbage Collector. Но ведь стандартные функции Kotlin (те же let, run, map, filter) — это тоже функции высшего порядка! Почему же тогда Kotlin не тормозит?"

- что это такое 
    - inline -- это модификтор, который ставится перед фунцкией. он говорит компилятору "не вызывай эту функцию как обычно, а просто скопируй весь ее код и вставь прямо в это место, откуда ее вызывают"

- зачем?
    - в прошлом вопросе мы затронули то, что функции высшего порядка создаются как объекты анонимного класса, из-за чего происходит большое потребление по памяти
    - и такой способ он как раз и решает эту проблему -- просто другой механиз передачи функционала, который не так пожирает память

- плюсы и минусы
    - плюсы
        - производительно 
        - reigied (ниже)
        - можно делать нелокальный возврат (non local return)
    - минусы
        - если метод -- 100 строк, а мест куда ты ее вставляешь -- 50, то в итоге получится очень много строк, тем самым увеличиться jar в размерах

- для чего использовать
    - когда делать 
        - когда небольшая функция высшего порядка
    - когда не делать 
        - для обычных функций, что без лямбд
        - типо -- JIT компиляция умеет сама инлайнить обычные функции

- вопросы 
    - что будет, если я напишу return внутри лямбды?
    ```kotlin
    inline fun doSmth(action: () -> Unit) { action() }

    fun test () {
        doSmth {
            println("lol")
            return 
        }
        println("Никогда не вызовится") // ибо код физически скопировался сюда
    }
    ```
    
    - ситуация -- inline функция принимает две лямбды. Одну я хочу выполнить сразу, а вторую — сохранить в переменную или передать в другой поток. Компилятор ругается. Почему? Далее решение
    ```kotlin
    inline fun process(action: () -> Unit, noinline onComplite: () -> Unit) {
        action() // скопируется
        val savedCallback = onComplite // станет объектом FunctionN и тем самым будет можно сохранить
    }
    ```

    - `crossinline` -- запрещает делать нелокальный возрват. чтобы не ломать логику в многопоточной среде

    - проблема стирания типов
        - типо
        ```kotlin
        // ПЛОХО
        fun <T> checkType(item: Any) {
            if (item is T) { // это не рабоатет
                ...
            }
        }
        ```
        - а если сделать ее inline и добавить reified (овеществленный), то все сработает 
        ```kotlin
        inline fun <reified T> checkType(item: Any) {
            if (item is T) {
                println("Оно")
            }
        }

        checkType<String>("Hello")
        ```
        - как это работает?
            - копирует тело вместо вызова
            - вместо T подставляет конкретный тип, который знает

- пример 
    - маппер JSON
    ```kotlin 
    inline fun<reified T> ObjectMapper.parse(json: String): T {
        return this.readValue(json. T::class.java)
    }

    val user: User = mapper.parse(json)
    ```

### Kotlin DSL (Domain Specific Language) (предметно ориентированный язык)

- что это   
    - подход, при котором ты пишешь код на Kotlin так, что он перестает быть походим на обычный код и становиться похожим на специализированный язык конфигурации (JSON, HTML, YAML), ну и при этом остается строго типизированным 
    - это называется Type-safe builders

- зачем?
    - описать сложные, вложенные иерархии объектов или конфигураций Декларативно (что хочу получить), а не императивно (шаг за шагом создать)

- плюсы и минусы

    - минусы
        - писать движок для DSL тяжело
        - легко переборщить. не нужно делать на каждый чих!

- для чего использовать
    - скрипты сборки gradle
    - роутинг в фреймворке Ktor
    - генерация HTML и XML
    - Jetpack Compose

### Ключевое слово tailrec

- что это?
    - ключевое слово, которое ставиться перед рекурсивной функцией. говорит компилятору "оптимизируй эту функцию так, чтобы она не переполняла стек вызовов"

- зачем?    
    - типичная ошибка переполнения стека вызывов. а это как ее решение

- плюсы
    - можно написать красивый рекурсивный код и не боятся проблем
- минусы
    - работает только если рекурсивная функция вызывается последней. иначе если есть что-то после -- не сработает

- приемры
    - математика
    - алгоритмы

- ловушка
    - пример
    ```kotlin
    tailrec fun factorial(n: Int): Long {
        if (n == 1) return 1
        // ОШИБКА ЗДЕСЬ:
        return n * factorial(n - 1)  // здесь последняя операция это умножение
    }

    tailrec fun factorial(n: Int, accumulator: Long = 1): Long {
        if (n == 1) return accumulator
        // Теперь вызов функции — это РЕАЛЬНО последнее действие!
        return factorial(n - 1, n * accumulator) 
    }
    ```
    - как под капотом
        - компилятор просто переписывает в обычный цикл 

- когда использовать 
    - обход дерева

### Иерархия и Мутабельность (Read-only vs Mutable)

- что это
    - они сделали List (неизменяемый, только методы чтения) и MutableList (неследует этот + методы для изменения). и так со всеми остальными (Set, Map)

- зачем?
    - чтобы на уровне компиляции защитить данные от случайных изменений

- плюсы и минусы
    - плюсы
        - безопастность на этапе написания кода
    - минусы
        - при компиляции -- это все тот же List. держится все "на слове"

- ловушки 
    - важно осознавать -- если мне и передают List, это не означает что он Неизменяемый (MutuableList наследуется от List между прочим. ничего не мешает мне Его передать)
    - Кастование (приведение типов) 
    ```kotlin
    fun process(list: List<String>) {
        if (list is MutableList) {
            list.add("Hack!") // может и упасть, а может и не упасть -- в зависимости как создавался (Arrays.asList() в Java все помнят)
        }
    }
    ```

- так же 
    - есть отдельная либа kotlinx.collections.immutable -- там есть ImmutableList, например
    - если хочешь поменять ImmutableList -- создаться новый объект, но там все достаточно эффективно -- структурное разделение памяти, как Git

- как использовать
    - желательно чтобы возвращал и принимал List. 

### Стандартные операции и производительность (map, filter, etc.)

- что это?
    - функции высшего порядка, что дают возможность обрабатывать коллекции в декларативном стиле
    - функции
        - map -- тип А в тип В
        - filter

- плюсы и минусы
    - плюсы
        - красиво
        - меньше шансов ошибиться в индексах
    - минусы
        - Жадные вычисления. по умолчанию на калекциях каждая операция выполняется немедленно и создает новую промежуточную коллекцию в памяти

### Iterable vs Sequence (Ленивые вычисления)

- что такое
    - Iterable -- стандартные жадные списки, что я указал выше. каждая операция, что описана выше, выполнялась для всех элементов сразу. это выходило в новый список
    - Sequence -- ленивый способ. не выполнятся, пока ты финальный резульатт не попросишь. проходят через всю цепочку по одному

- зачем?   
    - оптимизация 
        - никаких промежуточных коллекций
        - меньше работы 

- плюсы и минусы
    - плюсы
        - экономия памяти
        - Short-circuiting (Досрочный выход) -- если нужен только первый элемент -- мы найдем только первый и выйдем сразу
    - минусы
        - Overhead -- на малом объеме данных работает медленнее. ибо создание объектов-итераторов и управление ленивостью стоит дороже, чем просто пробежать по массиву

- ловушка 
    - стандартный прикол -- показывает кусок кода, а там нет терминальной операции. что выведет -- ничего
    - можно делать бесконечные последовательности
    ```kotlin 
    val numbers = generateSequence(1) { it + 1 }
    println(numbers.take(5).toList()) // [1,2,3,4,5]
    ```

- когда Sequence
    - использовать
        - большие объемы
        - длинные цепочки (от 3-4 шагов)
        - когда есть операции first(), find()...
    - не использовать
        - маленькие списки
        - если нужно просто один раз отфилтровать список
