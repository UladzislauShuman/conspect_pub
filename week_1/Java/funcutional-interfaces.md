## Что такое функциональный интерфейс? Примеры
это интерфейс, который определяет только один абстрактный метод

используется в основном в лямбда-выражениях и ссылке на метод (method reference)

так же важно -- только один Абстрактный метод. то есть, в интерфейсе может быть и несколько методов, просто главное что один абстрактный, а все остальные имеют default реализацию

## Для чего нужна аннотация @FunctionalInterface?
работает по такому же принципу, что и @Override.
чтобы точно определить интерфейс как функциональный.

т.е. -- она обозначает замысел и не даст определить второй абстрактный метод в интерфейсе

## Какие встроенные функциональные интерфейсы вы знаете?
- `Predicate<T>` -- для проверки соблюдения некоторого условия. в качестве параметра лямбда-выражения принимает объект типа T
- `Consumer<T>` -- выполняет действие над объектом Т, результат не возвращает
- `Function<T,R>` -- из T получить R
- `Supplier<T>` -- не принимает аргументов, возвращает объет типа Т
- `UnaryOperator<T>` -- параметр это объект Т. выполняет над ним операции и возвращает результат как объект Т.
- `BinaryOperator<T>` -- принимает как параметры два объекта Т. выполняет над ними бинарную операцию(сумма, умножение -- пример) И возвращает новый объект Т.

пример кода:
```java
Predicate<Integer> isPositive = x -> x > 0;
System.out.println(isPositive.test(5)); 

BinaryOperator<Integer> multiply = (x, y) -> x*y;
System.out.println(multiply.apply(3, 5));

UnaryOperator<Integer> square = x -> x*x;
System.out.println(square.apply(5)); 

Function<Integer, String> convert = x-> String.valueOf(x) + "долларов";
System.out.println(convert.apply(5));

Consumer<Integer> printer = x-> System.out.printf("%\dдолларов \n", x);
printer.accept(600); 

Supplier<User> userFactory = ()->{
    Scanner in = new Scanner(System.in);
    System.out.println("Введите имя: ");
    String name = in.nextLine();
    return new User(name);
};

User user1 = userFactory.get();
User user2 = userFactory.get();
```

## Что такое ссылка на метод?
Method References -- "компатные лямбда-выражения для методов, у которых уже есть названия"

виды ссылок:
- на статические методы: `ContainingClass::staticMethodName`
- на нестатические методы конкретного объекта: `containingObject::instanceMethodName`
- на нестатические методы любого объекта конкретного типа: `ContainingType::methodName`
- ссылка на конструктор: `ClassName::new`

от себя: в чем отличие тогда нестатического и статического, полученного через класс

примерно в том, как они под каппотом вызываются:
```java
List<Integer> numbers = strings.stream()
        .map(Integer::parseInt) // Ссылка на статический метод
        .collect(Collectors.toList());
// .map(s -> Integer.parseInt(s))
// параметр вставляется в метод статический

List<String> upperNames = names.stream()
        .map(String::toUpperCase) // Ссылка на метод произвольного объекта
        .collect(Collectors.toList());

// .map(name -> name.toUpperCase())
// у параметра вызывается метод, ссылку на который передали
```

и так же -- есть отличие между нестатическим методом, что получен через Класс и через Объект
```java
class Printer {
    public void print(String s) {
        System.out.println("Printing: " + s);
    }
}

Printer myPrinter = new Printer();
List<String> names = Arrays.asList("Ivan", "Maria");
names.forEach(myPrinter::print);
// (name) -> myPrinter.print(name)
// по сути, как у статического метода, просто мы теперь берем его из конретного объекта
```

## Что такое лямбда-выражение? Чем его можно заменить?

представляет набор инструкйий, которые можно выделить в отдельную переменнуюю и затем многократно вызвать в различных местах программы

как реализация метода, определенного в функциональном интерфейсе

`(список параметров выражение(соответствует по типу параметрам метода из функционального интерфейса)) -> тело лямбда-выражения`

в лямбда-выражениях нет generic-оф. должен быть конкретный тип

важное определение в контексте Лямбда-выражений == отложенное выполнение

deferred execution -- мы определяем в одном месте программы наше выражение, а затем можем его вызывать при нужде Неопределенное количество раз в разных местах программы

случаи, когда deferred execution может потребоваться
- выполнять код в отдельном потоке
- один и тот же код несколько раз выполнять
- выполнять код в результате события какого-то
- выполнять код только тогда, когда он реально необходим(lazy исполнение, то есть выполняем не в момент объявления, а когда оно реально надо) и если он необходим (выполнены условия какие-то)

Лямбда-выражение можно заменить Анонимным внутренним классом (Anonymous Inner Class).

```java
// Анонимный класс
new Thread(new Runnable() {
    @Override
    public void run() {
        System.out.println("Hello");
    }
}).start();

// Лямбда
new Thread(() -> System.out.println("Hello")).start();
```
важная и интересная разница 
- на что ссылается this:
    - В анонимном классе this ссылается на сам этот анонимный класс.
    - В лямбде this ссылается на внешний класс (где лямбда написана).
- Анонимный класс компилируется в отдельный .class файл (например, Main$1.class), а лямбда использует механизм invokedynamic и не плодит лишних файлов при компиляции.
