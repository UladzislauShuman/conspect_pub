## Что такое дженерики?

Это -- мезанизм в Java, который позволяет сделать классы/интерфейс/метод Который и работает с разными типами И при жтом не происходит потери типобезопастности

Generic типы/методы отличаются просто тем, что параметры типизированы.

Свойства 
- строгая типизация
- реализация едина
- у нас Нет информации о типе

Ограничения: 
в основном проблемы связаны с стиранием дженериков после компиляции \
что такое стирание?: это механизм, при котором во время компиляции информация о типах-параметрах удаляется, и в байткоде остаются обычные типы (чаще всего Object)

пример сигнатуры: MyGenericClass\<SomeType> (это все вместе называется обобщенным типом)

- в <> -- только ссылочные типы (кстати: int[] -- ссылочный, он подходит)
- внутри generic-класса/метода нельзя создать экземляр класса SomeType / его массив / использовать его equals (компилятору же неизвестно ничего о классе)
- пусть есть Foo extends Bar и некоторый G, который extends от коллекции. тогда G\<Foo> не наследуется от G\<Bar>
- наш параметр типа SomeType не может использоваться в "статическом контексте" (не может быть private static final параметром, или как-то в методе статическом использоваться.)
- Не может Создать, Поймать, или Бросить Объекты Параметризованных Типов. то есть
```java
class Box<T> {
    T value = new T(); // ошибка компиляции
}
```

```java
List<String>[] arr = new List<String>[10]; // ошибка

```

```java
class MyException<T> extends Exception {} // ошибка

try {
    ...
} catch (T e) { // ошибка
}

throw new T(); // ошибка
```

- У class не может быть двух перегруженных методов, у которых будет та же самая подпись после стирания типа.
```java
public class Example {

  public void print(Set<String> strSet) { }

  public void print(Set<Integer> intSet) { }
}
```

пример применения: Collections

создание объектов обобщенных типов происходит посредством замены параметризироаванных типов реальными типами данных 

## Для чего нужны Дженерики?
> Дженерики нужны для типобезопасности и переиспользования кода.
>
> До них обобщённый код писали через Object, из-за чего ошибки типов возникали только во время выполнения.
>
> Дженерики позволяют проверять типы на этапе компиляции и избавляют от явных приведений типов. В Java они реализованы через стирание типов.

От себя: по сути, у нас же и так была возможность сделать что-то обобщенное (к примереу -- параметр метода сделать Object и далее, так как все классы у нас наследуются от Object, можно будет подставлять любой объект/тип)

но -- это проверяется только в Runtime, что может быть опасно. (ошибки-типов обнаруживаются только во время работы программы). да и требовало явного приведения типов, что неудобно постоянно делать

Дженерики решают эту проблему:
- добавляют проверку типов на этапе компиляции;
- позволяют писать обобщённые классы, методы и структуры данных;
- избавляют от необходимости явных приведений типов.

В Java дженерики реализованы через стирание типов:
во время компиляции информация о типах-параметрах удаляется, а компилятор при необходимости автоматически добавляет приведения типов.

## Что такое сырые типы (raw type)?

это типы без указания типа параметра в <>

зачем? -- для поддержки старого кода

Type inference(вывод типов) -- когда компилятор сам допонимает, какой тип нужен был \
пример: 
```java
// тут String.                  а тут мы ничего не написали
List<String> list = new ArrayList<>();
// компилятор сможет Вывести тип
```
даже если не указаны <>, то компилятор выведит тип, но -- это смесь старого и нового стилей, и тем самым мы потеряем типобезопастость. \
то есть -- можно добавить какой угодно тип в список
```java
ArrayList<String> strings = new ArrayList<>(); // parameterized type
ArrayList arrayList = new ArrayList(); // raw type
arrayList = strings; // Ok
strings = arrayList; // Unchecked assignment (назначение)
arrayList.add(1); //unchecked call
```

## Что такое вайлдкарды (Маски)?

это конструкция внутри <> (алмазный-оператор), что позволяет сделать код более универсальным

помогает решить проблему наследования (когда G\<Foo> не наследуется от G\<Bar>)

рассмотрим иерархию:
Cat/Dog -> Pet -> Animal -> Object

вайлдкарды могут быть трех типов
- Инвариантность (Когда мы явно указываем тип(List\<String>))
- Upper (<? extends Animal> -- ? = любой наследник Animal. то есть -- вверх идет)
- Lower (<? super Cat> -- ? = любой Родитель Cat И включая Cat. т.е. -- идем вниз по родословной)

Пример для extends
```java
// Объявление списка с wildcard (джокером)
List<? extends Number> numberList = new ArrayList<>();

List<Integer> integerList = new ArrayList<>();
numberList = integerList; // Это работает! Мы можем присвоить список наследников

integerList.add(5); // Это работает, так как integerList знает, что он хранит Integer
numberList.add(5);  // ОШИБКА КОМПИЛЯЦИИ!

// Пример в методе
public void process(List<? extends Number> list) {
     list.add(5L); // ОШИБКА КОМПИЛЯЦИИ
     // Мы не можем ничего положить, кроме null
}
```

Пример для super
```java
List<? super Number> numberList = new ArrayList<>();

// Сюда можно добавлять Number и его наследников
numberList.add(10);       // Integer (ок)
numberList.add(10.5);     // Double (ок)
numberList.add(5L);       // Long (ок)

// Но нельзя добавить родителя (Object)
// numberList.add(new Object()); // Ошибка компиляции
```

| Характеристика | `List<? extends Number>` (Upper Bound) | `List<? super Number>` (Lower Bound) |
| :--- | :--- | :--- |
| **Что это значит?** | Ссылка на список **конкретного, но неизвестного подтипа** `Number`. <br>*(Это может быть `List<Integer>`, `List<Double>` и т.д.)* | Ссылка на список **конкретного, но неизвестного родителя** `Number`. <br>*(Это может быть `List<Number>` или `List<Object>`)* |
| **Операция чтения (`get`)** | **✅ Гарантировано.** <br>Мы получаем `Number`. Мы точно знаем, что любой элемент там — это число. | **⚠️ Бесполезно (почти).** <br>Мы получаем `Object`. Мы не знаем, число это или строка, поэтому теряем типизацию. |
| **Операция записи (`add`)** | **❌ ЗАПРЕЩЕНО.** <br>Нельзя добавить ничего (кроме `null`). Компилятор блокирует любые попытки вставки. | **✅ РАЗРЕШЕНО.** <br>Можно добавлять `Number` и любые его подтипы (`Integer`, `Double`, `Long`). |
| **Философия (Почему так?)** | **Проблема:** Мы не знаем, на какой именно список указывает ссылка. <br>Если ссылка смотрит на `List<Integer>`, а мы попытаемся вставить `Double` — мы сломаем список. Чтобы этого не допустить, Java запрещает писать вообще. | **Логика:** Мы знаем, что список умеет хранить как минимум `Number` (или его родителей типа `Object`). <br>`Integer` является и `Number`, и `Object`. Значит, безопасно положить `Integer` в любой из этих списков. |
| **Роль (PECS)** | **Producer (Поставщик).** <br>Список поставляет нам данные для чтения. | **Consumer (Потребитель).** <br>Список потребляет наши данные (мы в него пишем). |
| **Бизнес-задача (Пример)** | **Аналитика:** Метод, который принимает список любых чисел (неважно, целых или дробных) и считает их среднее арифметическое. | **Заполнение отчета:** Метод, который принимает список-накопитель (неважно, чисто для чисел или для всех объектов) и дописывает туда новые финансовые показатели. |

## Расскажите про принцип PECS

**PECS** — это правило работы с wildcard-типами в дженериках Java:

* **Producer Extends** — если коллекция **производит (возвращает)** элементы, используем `? extends T`.
* **Consumer Super** — если коллекция **потребляет (принимает)** элементы, используем `? super T`.


### Producer — `? extends T` (ковариантность)

```java
List<? extends Animal>
```

* Коллекция может содержать `Animal` или любой его подкласс.
* Из неё **безопасно читать** элементы как `Animal`.
* **Нельзя добавлять** новые элементы (кроме `null`).

Это называется **ковариантностью** — тип сохраняет направление наследования.

### Consumer — `? super T` (контрвариантность)

```java
List<? super Cat>
```

* Коллекция может быть `List<Cat>`, `List<Animal>` или `List<Object>`.
* В неё **можно добавлять** объекты типа `Cat` и его подклассов.
* При чтении элементы имеют тип `Object`.

Это называется **контрвариантностью** — направление наследования «обращается».

### Неограниченный wildcard

```java
Collection<?>
```

* То же самое, что `Collection<? extends Object>`.
* **Нельзя ничего добавлять** (кроме `null`).
* Можно только читать элементы как `Object`.
